3    :mod:`pygame.Surface`
4    =====================
5    
6    .. currentmodule:: pygame
7    
8    .. class:: Surface
9    
10      | :sl:`pygame object for representing images`
11      | :sg:`Surface((width, height), flags=0, depth=0, masks=None) -> Surface`
12      | :sg:`Surface((width, height), flags=0, Surface) -> Surface`
13   
14      A pygame Surface is used to represent any image. The Surface has a fixed
15      resolution and pixel format. Surfaces with 8-bit pixels use a color palette
16      to map to 24-bit color.
17   
18      Call :meth:`pygame.Surface()` to create a new image object. The Surface will
19      be cleared to all black. The only required arguments are the sizes. With no
20      additional arguments, the Surface will be created in a format that best
21      matches the display Surface.
22   
23      The pixel format can be controlled by passing the bit depth or an existing
24      Surface. The flags argument is a bitmask of additional features for the
25      surface. You can pass any combination of these flags:
26   
27      ::
28   
29        HWSURFACE    (obsolete in pygame 2) creates the image in video memory
30        SRCALPHA     the pixel format will include a per-pixel alpha
31   
32      Both flags are only a request, and may not be possible for all displays and
33      formats.
34   
35      Advance users can combine a set of bitmasks with a depth value. The masks
36      are a set of 4 integers representing which bits in a pixel will represent
37      each color. Normal Surfaces should not require the masks argument.
38   
39      Surfaces can have many extra attributes like alpha planes, colorkeys, source
40      rectangle clipping. These functions mainly effect how the Surface is blitted
41      to other Surfaces. The blit routines will attempt to use hardware
42      acceleration when possible, otherwise they will use highly optimized
43      software blitting methods.
44   
45      There are three types of transparency supported in pygame: colorkeys,
46      surface alphas, and pixel alphas. Surface alphas can be mixed with
47      colorkeys, but an image with per pixel alphas cannot use the other modes.
48      Colorkey transparency makes a single color value transparent. Any pixels
49      matching the colorkey will not be drawn. The surface alpha value is a single
50      value that changes the transparency for the entire image. A surface alpha of
51      255 is opaque, and a value of 0 is completely transparent.
52   
53      Per pixel alphas are different because they store a transparency value for
54      every pixel. This allows for the most precise transparency effects, but it
55      also the slowest. Per pixel alphas cannot be mixed with surface alpha and
56      colorkeys.
57   
58      There is support for pixel access for the Surfaces. Pixel access on hardware
59      surfaces is slow and not recommended. Pixels can be accessed using the
60      :meth:`get_at()` and :meth:`set_at()` functions. These methods are fine for
61      simple access, but will be considerably slow when doing of pixel work with
62      them. If you plan on doing a lot of pixel level work, it is recommended to
63      use a :class:`pygame.PixelArray`, which gives an array like view of the
64      surface. For involved mathematical manipulations try the
65      :mod:`pygame.surfarray` module (It's quite quick, but requires NumPy.)
66   
67      Any functions that directly access a surface's pixel data will need that
68      surface to be lock()'ed. These functions can :meth:`lock()` and
69      :meth:`unlock()` the surfaces themselves without assistance. But, if a
70      function will be called many times, there will be a lot of overhead for
71      multiple locking and unlocking of the surface. It is best to lock the
72      surface manually before making the function call many times, and then
73      unlocking when you are finished. All functions that need a locked surface
74      will say so in their docs. Remember to leave the Surface locked only while
75      necessary.
76   
77      Surface pixels are stored internally as a single number that has all the
78      colors encoded into it. Use the :meth:`map_rgb()` and
79      :meth:`unmap_rgb()` to convert between individual red, green, and blue
80      values into a packed integer for that Surface.
81   
82      Surfaces can also reference sections of other Surfaces. These are created
83      with the :meth:`subsurface()` method. Any change to either Surface will
84      effect the other.
85   
86      Each Surface contains a clipping area. By default the clip area covers the
87      entire Surface. If it is changed, all drawing operations will only effect
88      the smaller area.

90      .. method:: blit
91   
92         | :sl:`draw one image onto another`
93         | :sg:`blit(source, dest, area=None, special_flags=0) -> Rect`
94   
95         Draws a source Surface onto this Surface. The draw can be positioned with
96         the dest argument. The dest argument can either be a pair of coordinates representing the position of
97         the upper left corner of the blit or a Rect, where the upper left corner of the rectangle will be used as the
98         position for the blit. The size of the destination rectangle does not
99         effect the blit.
100  
101        An optional area rectangle can be passed as well. This represents a
102        smaller portion of the source Surface to draw.
103  
104        .. versionadded:: 1.8
105           Optional ``special_flags``: ``BLEND_ADD``, ``BLEND_SUB``,
106           ``BLEND_MULT``, ``BLEND_MIN``, ``BLEND_MAX``.
107  
108        .. versionadded:: 1.8.1
109           Optional ``special_flags``: ``BLEND_RGBA_ADD``, ``BLEND_RGBA_SUB``,
110           ``BLEND_RGBA_MULT``, ``BLEND_RGBA_MIN``, ``BLEND_RGBA_MAX``
111           ``BLEND_RGB_ADD``, ``BLEND_RGB_SUB``, ``BLEND_RGB_MULT``,
112           ``BLEND_RGB_MIN``, ``BLEND_RGB_MAX``.
113  
114        .. versionadded:: 1.9.2
115           Optional ``special_flags``: ``BLEND_PREMULTIPLIED``
116  
117        .. versionadded:: 2.0.0
118           Optional ``special_flags``:  ``BLEND_ALPHA_SDL2`` - Uses the SDL2 blitter for alpha blending,
119           this gives different results than the default blitter, which is modelled after SDL1, due to
120           different approximations used for the alpha blending formula. The SDL2 blitter also supports
121           RLE on alpha blended surfaces which the pygame one does not.
122  
123        The return rectangle is the area of the affected pixels, excluding any
124        pixels outside the destination Surface, or outside the clipping area.
125  
126        Pixel alphas will be ignored when blitting to an 8 bit Surface.
127  
128        For a surface with colorkey or blanket alpha, a blit to self may give
129        slightly different colors than a non self-blit.

133     .. method:: blits
134  
135        | :sl:`draw many images onto another`
136        | :sg:`blits(blit_sequence=((source, dest), ...), doreturn=1) -> [Rect, ...] or None`
137        | :sg:`blits(((source, dest, area), ...)) -> [Rect, ...]`
138        | :sg:`blits(((source, dest, area, special_flags), ...)) -> [Rect, ...]`
139  
140        Draws many surfaces onto this Surface. It takes a sequence as input,
141        with each of the elements corresponding to the ones of :meth:`blit()`.
142        It needs at minimum a sequence of (source, dest).
143  
144        :param blit_sequence: a sequence of surfaces and arguments to blit them,
145           they correspond to the :meth:`blit()` arguments
146        :param doreturn: if ``True``, return a list of rects of the areas changed,
147           otherwise return ``None``
148  
149        :returns: a list of rects of the areas changed if ``doreturn`` is
150           ``True``, otherwise ``None``
151        :rtype: list or None
152  
153        New in pygame 1.9.4.

158     .. method:: convert
159  
160        | :sl:`change the pixel format of an image`
161        | :sg:`convert(Surface=None) -> Surface`
162        | :sg:`convert(depth, flags=0) -> Surface`
163        | :sg:`convert(masks, flags=0) -> Surface`
164  
165        Creates a new copy of the Surface with the pixel format changed. The new
166        pixel format can be determined from another existing Surface. Otherwise
167        depth, flags, and masks arguments can be used, similar to the
168        :meth:`pygame.Surface()` call.
169  
170        If no arguments are passed the new Surface will have the same pixel
171        format as the display Surface. This is always the fastest format for
172        blitting. It is a good idea to convert all Surfaces before they are
173        blitted many times.
174  
175        The converted Surface will have no pixel alphas. They will be stripped if
176        the original had them. See :meth:`convert_alpha()` for preserving or
177        creating per-pixel alphas.
178  
179        The new copy will have the same class as the copied surface. This lets
180        as Surface subclass inherit this method without the need to override,
181        unless subclass specific instance attributes also need copying.
182  
183        .. ## Surface.convert ##
184  
185     .. method:: convert_alpha
186  
187        | :sl:`change the pixel format of an image including per pixel alphas`
188        | :sg:`convert_alpha(Surface) -> Surface`
189        | :sg:`convert_alpha() -> Surface`
190  
191        Creates a new copy of the surface with the desired pixel format. The new
192        surface will be in a format suited for quick blitting to the given format
193        with per pixel alpha. If no surface is given, the new surface will be
194        optimized for blitting to the current display.
195  
196        Unlike the :meth:`convert()` method, the pixel format for the new
197        image will not be exactly the same as the requested source, but it will
198        be optimized for fast alpha blitting to the destination.
199  
200        As with :meth:`convert()` the returned surface has the same class as
201        the converted surface.

221  
222        Fill the Surface with a solid color. If no rect argument is given the

229        the ``SRCALPHA`` flag).

260     .. method:: set_colorkey
261  
262        | :sl:`Set the transparent colorkey`
263        | :sg:`set_colorkey(Color, flags=0) -> None`
264        | :sg:`set_colorkey(None) -> None`
265  
266        Set the current color key for the Surface. When blitting this Surface
267        onto a destination, any pixels that have the same color as the colorkey
268        will be transparent. The color can be an ``RGB`` color or a mapped color
269        integer. If ``None`` is passed, the colorkey will be unset.
270  
271        The colorkey will be ignored if the Surface is formatted to use per pixel
272        alpha values. The colorkey can be mixed with the full Surface alpha
273        value.
274  
275        The optional flags argument can be set to ``pygame.RLEACCEL`` to provide
276        better performance on non accelerated displays. An ``RLEACCEL`` Surface
277        will be slower to modify, but quicker to blit as a source.

324     .. method:: lock
325  
326        | :sl:`lock the Surface memory for pixel access`
327        | :sg:`lock() -> None`
328  
329        Lock the pixel data of a Surface for access. On accelerated Surfaces, the
330        pixel data may be stored in volatile video memory or nonlinear compressed
331        forms. When a Surface is locked the pixel memory becomes available to
332        access by regular software. Code that reads or writes pixel values will
333        need the Surface to be locked.
334  
335        Surfaces should not remain locked for more than necessary. A locked
336        Surface can often not be displayed or managed by pygame.
337  
338        Not all Surfaces require locking. The :meth:`mustlock()` method can
339        determine if it is actually required. There is no performance penalty for
340        locking and unlocking a Surface that does not need it.
341  
342        All pygame functions will automatically lock and unlock the Surface data
343        as needed. If a section of code is going to make calls that will
344        repeatedly lock and unlock the Surface many times, it can be helpful to
345        wrap the block inside a lock and unlock pair.
346  
347        It is safe to nest locking and unlocking calls. The surface will only be
348        unlocked after the final lock is released.
349

407     .. method:: get_at
408  
409        | :sl:`get the color value at a single pixel`
410        | :sg:`get_at((x, y)) -> Color`
411  
412        Return a copy of the ``RGBA`` Color value at the given pixel. If the
413        Surface has no per pixel alpha, then the alpha value will always be 255
414        (opaque). If the pixel position is outside the area of the Surface an
415        ``IndexError`` exception will be raised.
416  
417        Getting and setting pixels one at a time is generally too slow to be used
418        in a game or realtime situation. It is better to use methods which
419        operate on many pixels at a time like with the blit, fill and draw
420        methods - or by using :mod:`pygame.surfarray`/:mod:`pygame.PixelArray`.
421  
422        This function will temporarily lock and unlock the Surface as needed.
423  
424        .. versionadded:: 1.9
425           Returning a Color instead of tuple. Use ``tuple(surf.get_at((x,y)))``
426           if you want a tuple, and not a Color. This should only matter if
427           you want to use the color as a key in a dict.

577     .. method:: subsurface
578  
579        | :sl:`create a new surface that references its parent`
580        | :sg:`subsurface(Rect) -> Surface`
581  
582        Returns a new Surface that shares its pixels with its new parent. The new
583        Surface is considered a child of the original. Modifications to either
584        Surface pixels will effect each other. Surface information like clipping
585        area and color keys are unique to each Surface.
586  
587        The new Surface will inherit the palette, color key, and alpha settings
588        from its parent.
589  
590        It is possible to have any number of subsurfaces and subsubsurfaces on
591        the parent. It is also possible to subsurface the display Surface if the
592        display mode is not hardware accelerated.
593  
594        See :meth:`get_offset()` and :meth:`get_parent()` to learn more
595        about the state of a subsurface.
596  
597        A subsurface will have the same class as the parent surface.

829     .. method:: get_view
830  
831        | :sl:`return a buffer view of the Surface's pixels.`
832        | :sg:`get_view(<kind>='2') -> BufferProxy`
833  
834        Return an object which exports a surface's internal pixel buffer as
835        a C level array struct, Python level array interface or a C level
836        buffer interface. The new buffer protocol is supported.
837  
838        The kind argument is the length 1 string '0', '1', '2', '3',
839        'r', 'g', 'b', or 'a'. The letters are case insensitive;
840        'A' will work as well. The argument can be either a Unicode or byte (char)
841        string. The default is '2'.
842  
843        '0' returns a contiguous unstructured bytes view. No surface shape
844        information is given. A ``ValueError`` is raised if the surface's pixels
845        are discontinuous.
846  
847        '1' returns a (surface-width * surface-height) array of continuous
848        pixels. A ``ValueError`` is raised if the surface pixels are
849        discontinuous.
850  
851        '2' returns a (surface-width, surface-height) array of raw pixels.
852        The pixels are surface-bytesize-d unsigned integers. The pixel format is
853        surface specific. The 3 byte unsigned integers of 24 bit surfaces are
854        unlikely accepted by anything other than other pygame functions.
855  
856        '3' returns a (surface-width, surface-height, 3) array of ``RGB`` color
857        components. Each of the red, green, and blue components are unsigned
858        bytes. Only 24-bit and 32-bit surfaces are supported. The color
859        components must be in either ``RGB`` or ``BGR`` order within the pixel.
860  
861        'r' for red, 'g' for green, 'b' for blue, and 'a' for alpha return a
862        (surface-width, surface-height) view of a single color component within a
863        surface: a color plane. Color components are unsigned bytes. Both 24-bit
864        and 32-bit surfaces support 'r', 'g', and 'b'. Only 32-bit surfaces with
865        ``SRCALPHA`` support 'a'.
866  
867        The surface is locked only when an exposed interface is accessed.
868        For new buffer interface accesses, the surface is unlocked once the
869        last buffer view is released. For array interface and old buffer
870        interface accesses, the surface remains locked until the BufferProxy
871        object is released.
872  
873        .. versionadded:: 1.9.2
874  
875     .. method:: get_buffer
876  
877        | :sl:`acquires a buffer object for the pixels of the Surface.`
878        | :sg:`get_buffer() -> BufferProxy`
879  
880        Return a buffer object for the pixels of the Surface. The buffer can be
881        used for direct pixel access and manipulation. Surface pixel data is
882        represented as an unstructured block of memory, with a start address
883        and length in bytes. The data need not be contiguous. Any gaps are
884        included in the length, but otherwise ignored.
885  
886        This method implicitly locks the Surface. The lock will be released when
887        the returned :mod:`pygame.BufferProxy` object is garbage collected.
888
