100  <section id="sprite-module-introduction">
101  <h2>Sprite Module Introduction<a class="headerlink" href="#sprite-module-introduction" title="Permalink to this headline">¶</a></h2>
102  <dl class="docinfo field-list simple">
103  <dt class="field-odd">Author</dt>
104  <dd class="field-odd"><p>Pete Shinners</p>
105  </dd>
106  <dt class="field-even">Contact</dt>
107  <dd class="field-even"><p><a class="reference external" href="mailto:pete&#37;&#52;&#48;shinners&#46;org">pete<span>&#64;</span>shinners<span>&#46;</span>org</a></p>
108  </dd>
109  </dl>
110  <p>Pygame version 1.3 comes with a new module, <code class="docutils literal notranslate"><span class="pre">pygame.sprite</span></code>. This module is
111  written in Python and includes some higher-level classes to manage your game
112  objects. By using this module to its full potential, you can easily manage and
113  draw your game objects. The sprite classes are very optimized, so it's likely
114  your game will run faster with the sprite module than without.</p>
115  <p>The sprite module is also meant to be very generic. It turns out you can use it
116  with nearly any type of gameplay. All this flexibility comes with a slight
117  penalty, it needs a little understanding to properly use it. The
118  <a class="reference internal" href="../ref/sprite.html#module-pygame.sprite" title="pygame.sprite: pygame module with basic game object classes"><code class="xref py py-mod docutils literal notranslate"><span class="pre">reference</span> <span class="pre">documentation</span></code></a> for the sprite module can keep
119  you running, but you'll probably need a bit more explanation of how to use
120  <code class="docutils literal notranslate"><span class="pre">pygame.sprite</span></code> in your own game.</p>
121  <p>Several of the pygame examples (like &quot;chimp&quot; and &quot;aliens&quot;) have been updated to
122  use the sprite module. You may want to look into those first to see what this
123  sprite module is all about. The chimp module even has its own line-by-line
124  tutorial, which may help get more an understanding of programming with python
125  and pygame.</p>
126  <p>Note that this introduction will assume you have a bit of experience
127  programming with python, and are somewhat familiar with the different parts of
128  creating a simple game.  In this tutorial the word &quot;reference&quot; is occasionally
129  used.  This represents a python variable. Variables in python are references,
130  so you can have several variables all pointing to the same object.</p>
131  <section id="history-lesson">
132  <h3>History Lesson<a class="headerlink" href="#history-lesson" title="Permalink to this headline">¶</a></h3>
133  <p>The term &quot;sprite&quot; is a holdover from older computer and game machines.  These
134  older boxes were unable to draw and erase normal graphics fast enough for them
135  to work as games. These machines had special hardware to handle game like
136  objects that needed to animate very quickly. These objects were called
137  &quot;sprites&quot; and had special limitations, but could be drawn and updated very
138  fast. They usually existed in special overlay buffers in the video.  These days
139  computers have become generally fast enough to handle sprite like objects
140  without dedicated hardware. The term sprite is still used to represent just
141  about anything in a 2D game that is animated.</p>
142  </section>
143  <section id="the-classes">
144  <h3>The Classes<a class="headerlink" href="#the-classes" title="Permalink to this headline">¶</a></h3>
145  <p>The sprite module comes with two main classes. The first is <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Sprite" title="pygame.sprite.Sprite"><code class="xref py py-class docutils literal notranslate"><span class="pre">Sprite</span></code></a>, which should be used as a base class for all your game
146  objects. This class doesn't really do anything on its own, it just includes
147  several functions to help manage the game object. The other type of class is
148  <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group" title="pygame.sprite.Group"><code class="xref py py-class docutils literal notranslate"><span class="pre">Group</span></code></a>. The <code class="docutils literal notranslate"><span class="pre">Group</span></code> class is a container for
149  different <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> objects. There are actually several different types of
150  group classes. Some of the <code class="docutils literal notranslate"><span class="pre">Groups</span></code> can draw all the elements they contain,
151  for example.</p>
152  <p>This is all there really is to it. We'll start with a description of what each
153  type of class does, and then discuss the proper ways to use these two classes.</p>
154  </section>
155  <section id="the-sprite-class">
156  <h3>The Sprite Class<a class="headerlink" href="#the-sprite-class" title="Permalink to this headline">¶</a></h3>
157  <p>As mentioned before, the Sprite class is designed to be a base class for all
158  your game objects. You cannot really use it on its own, as it only has several
159  methods to help it work with the different <code class="docutils literal notranslate"><span class="pre">Group</span></code> classes. The sprite keeps
160  track of which groups it belongs to.
161  The class constructor (<code class="docutils literal notranslate"><span class="pre">__init__</span></code> method) takes an argument of a
162  <code class="docutils literal notranslate"><span class="pre">Group</span></code> (or list of <code class="docutils literal notranslate"><span class="pre">Groups</span></code>) the <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> instance should belong to.
163  You can also change the <code class="docutils literal notranslate"><span class="pre">Group</span></code> membership for the <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> with the
164  <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Sprite.add" title="pygame.sprite.Sprite.add"><code class="xref py py-meth docutils literal notranslate"><span class="pre">add()</span></code></a> and
165  <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Sprite.remove" title="pygame.sprite.Sprite.remove"><code class="xref py py-meth docutils literal notranslate"><span class="pre">remove()</span></code></a> methods.
166  There is also a <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Sprite.groups" title="pygame.sprite.Sprite.groups"><code class="xref py py-meth docutils literal notranslate"><span class="pre">groups()</span></code></a> method,
167  which returns a list of the current groups containing the sprite.</p>
168  <p>When using the your Sprite classes it's best to think of them as &quot;valid&quot; or
169  &quot;alive&quot; when they are belonging to one or more <code class="docutils literal notranslate"><span class="pre">Groups</span></code>. When you remove the
170  instance from all groups pygame will clean up the object. (Unless you have your
171  own references to the instance somewhere else.) The <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Sprite.kill" title="pygame.sprite.Sprite.kill"><code class="xref py py-meth docutils literal notranslate"><span class="pre">kill()</span></code></a> method removes the sprite from all groups it
172  belongs to. This will cleanly delete the sprite object. If you've put some
173  little games together, you'll know sometimes cleanly deleting a game object can
174  be tricky. The sprite also comes with an <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Sprite.alive" title="pygame.sprite.Sprite.alive"><code class="xref py py-meth docutils literal notranslate"><span class="pre">alive()</span></code></a> method, which returns true if it is still a
175  member of any groups.</p>
176  </section>
177  <section id="the-group-class">
178  <h3>The Group Class<a class="headerlink" href="#the-group-class" title="Permalink to this headline">¶</a></h3>
179  <p>The <code class="docutils literal notranslate"><span class="pre">Group</span></code> class is just a simple container. Similar to the sprite, it has
180  an <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.add" title="pygame.sprite.Group.add"><code class="xref py py-meth docutils literal notranslate"><span class="pre">add()</span></code></a> and <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.remove" title="pygame.sprite.Group.remove"><code class="xref py py-meth docutils literal notranslate"><span class="pre">remove()</span></code></a> method which can change which sprites belong to
181  the group. You also can pass a sprite or list of sprites to the constructor
182  (<code class="docutils literal notranslate"><span class="pre">__init__()</span></code> method) to create a <code class="docutils literal notranslate"><span class="pre">Group</span></code> instance that contains some
183  initial sprites.</p>
184  <p>The <code class="docutils literal notranslate"><span class="pre">Group</span></code> has a few other methods like <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.empty" title="pygame.sprite.Group.empty"><code class="xref py py-meth docutils literal notranslate"><span class="pre">empty()</span></code></a> to remove all sprites from the group and
185  <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.copy" title="pygame.sprite.Group.copy"><code class="xref py py-meth docutils literal notranslate"><span class="pre">copy()</span></code></a> which will return a copy of the group
186  with all the same members. Also the <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.has" title="pygame.sprite.Group.has"><code class="xref py py-meth docutils literal notranslate"><span class="pre">has()</span></code></a>
187  method will quickly check if the <code class="docutils literal notranslate"><span class="pre">Group</span></code> contains a sprite or list of
188  sprites.</p>
189  <p>The other function you will use frequently is the <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.sprites" title="pygame.sprite.Group.sprites"><code class="xref py py-meth docutils literal notranslate"><span class="pre">sprites()</span></code></a> method. This returns an object that can be
190  looped on to access every sprite the group contains.  Currently this is just a
191  list of the sprites, but in later version of python this will likely use
192  iterators for better performance.</p>
193  <p>As a shortcut, the <code class="docutils literal notranslate"><span class="pre">Group</span></code> also has an <a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group.update" title="pygame.sprite.Group.update"><code class="xref py py-meth docutils literal notranslate"><span class="pre">update()</span></code></a> method, which will call an <code class="docutils literal notranslate"><span class="pre">update()</span></code> method on
194  every sprite in the group. Passing the same arguments to each one. Usually in a
195  game you need some function that updates the state of a game object. It's very
196  easy to call your own methods using the <code class="docutils literal notranslate"><span class="pre">Group.sprites()</span></code> method, but this is
197  a shortcut that's used enough to be included. Also note that the base
198  <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> class has a &quot;dummy&quot; <code class="docutils literal notranslate"><span class="pre">update()</span></code> method that takes any sort of
199  arguments and does nothing.</p>
200  <p>Lastly, the Group has a couple other methods that allow you to use it with
201  the builtin <code class="docutils literal notranslate"><span class="pre">len()</span></code> function, getting the number of sprites it contains, and
202  the &quot;truth&quot; operator, which allows you to do &quot;if mygroup:&quot; to check if the
203  group has any sprites.</p>
204  </section>
205  <section id="mixing-them-together">
206  <h3>Mixing Them Together<a class="headerlink" href="#mixing-them-together" title="Permalink to this headline">¶</a></h3>
207  <p>At this point the two classes seem pretty basic. Not doing a lot more than you
208  can do with a simple list and your own class of game objects. But there are
209  some big advantages to using the <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> and <code class="docutils literal notranslate"><span class="pre">Group</span></code> together. A sprite
210  can belong to as many groups as you want. Remember as soon as it belongs to no
211  groups, it will usually be cleared up (unless you have other &quot;non-group&quot;
212  references to that object).</p>
213  <p>The first big thing is a fast simple way to categorize sprites. For example,
214  say we had a Pacman-like game. We could make separate groups for the different
215  types of objects in the game. Ghosts, Pac, and Pellets. When Pac eats a power
216  pellet, we can change the state for all ghost objects by effecting everything
217  in the Ghost group. This is quicker and simpler than looping through a list
218  of all the game objects and checking which ones are ghosts.</p>
219  <p>Adding and removing groups and sprites from each other is a very fast
220  operation, quicker than using lists to store everything. Therefore you can very
221  efficiently change group memberships. Groups can be used to work like simple
222  attributes for each game object. Instead of tracking some attribute like
223  &quot;close_to_player&quot; for a bunch of enemy objects, you could add them to a
224  separate group. Then when you need to access all the enemies that are near the
225  player, you already have a list of them, instead of going through a list of all
226  the enemies, checking for the &quot;close_to_player&quot; flag. Later on your game could
227  add multiple players, and instead of adding more &quot;close_to_player2&quot;,
228  &quot;close_to_player3&quot; attributes, you can easily add them to different groups or
229  each player.</p>
230  <p>Another important benefit of using the <code class="docutils literal notranslate"><span class="pre">Sprites</span></code> and <code class="docutils literal notranslate"><span class="pre">Groups</span></code> is that the groups
231  cleanly handle the deleting (or killing) of game objects. In a game where many
232  objects are referencing other objects, sometimes deleting an object can be the
233  hardest part, since it can't go away until it is not referenced by anyone. Say
234  we have an object that is &quot;chasing&quot; another object. The chaser can keep a
235  simple Group that references the object (or objects) it is chasing. If the
236  object being chased happens to be destroyed, we don't need to worry about
237  notifying the chaser to stop chasing. The chaser can see for itself that its
238  group is now empty, and perhaps find a new target.</p>
239  <p>Again, the thing to remember is that adding and removing sprites from groups is
240  a very cheap/fast operation. You may be best off by adding many groups to
241  contain and organize your game objects. Some could even be empty for large
242  portions of the game, there isn't any penalties for managing your game like
243  this.</p>
244  </section>
245  <section id="the-many-group-types">
246  <h3>The Many Group Types<a class="headerlink" href="#the-many-group-types" title="Permalink to this headline">¶</a></h3>
247  <p>The above examples and reasons to use <code class="docutils literal notranslate"><span class="pre">Sprites</span></code> and <code class="docutils literal notranslate"><span class="pre">Groups</span></code> are only a tip
248  of the iceberg. Another advantage is that the sprite module comes with several
249  different types of <code class="docutils literal notranslate"><span class="pre">Groups</span></code>. These groups all work just like a regular old
250  <code class="docutils literal notranslate"><span class="pre">Group</span></code>, but they also have added functionality (or slightly different
251  functionality).  Here's a list of the <code class="docutils literal notranslate"><span class="pre">Group</span></code> classes included with the
252  sprite module.</p>
253  <blockquote>
254  <div><p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.Group" title="pygame.sprite.Group"><code class="xref py py-class docutils literal notranslate"><span class="pre">Group</span></code></a></p>
255  <blockquote>
256  <div><p>This is the standard &quot;no frills&quot; group mainly explained above. Most of the
257  other <code class="docutils literal notranslate"><span class="pre">Groups</span></code> are derived from this one, but not all.</p>
258  </div></blockquote>
259  <p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.GroupSingle" title="pygame.sprite.GroupSingle"><code class="xref py py-class docutils literal notranslate"><span class="pre">GroupSingle</span></code></a></p>
260  <blockquote>
261  <div><p>This works exactly like the regular <code class="docutils literal notranslate"><span class="pre">Group</span></code> class, but it only contains
262  the most recently added sprite. Therefore when you add a sprite to this group,
263  it &quot;forgets&quot; about any previous sprites it had. Therefore it always contains
264  only one or zero sprites.</p>
265  </div></blockquote>
266  <p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.RenderPlain" title="pygame.sprite.RenderPlain"><code class="xref py py-class docutils literal notranslate"><span class="pre">RenderPlain</span></code></a></p>
267  <blockquote>
268  <div><p>This is a standard group derived from <code class="docutils literal notranslate"><span class="pre">Group</span></code>. It has a draw() method
269  that draws all the sprites it contains to the screen (or any <code class="docutils literal notranslate"><span class="pre">Surface</span></code>). For
270  this to work, it requires all sprites it contains to have a &quot;image&quot; and &quot;rect&quot;
271  attributes. It uses these to know what to blit, and where to blit it.</p>
272  </div></blockquote>
273  <p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.RenderClear" title="pygame.sprite.RenderClear"><code class="xref py py-class docutils literal notranslate"><span class="pre">RenderClear</span></code></a></p>
274  <blockquote>
275  <div><p>This is derived from the <code class="docutils literal notranslate"><span class="pre">RenderPlain</span></code> group, and adds a method named
276  <code class="docutils literal notranslate"><span class="pre">clear()</span></code>. This will erase the previous position of all drawn sprites. It
277  uses a background image to fill in the areas where the sprite were. It is smart
278  enough to handle deleted sprites and properly clear them from the screen when
279  the <code class="docutils literal notranslate"><span class="pre">clear()</span></code> method is called.</p>
280  </div></blockquote>
281  <p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.RenderUpdates" title="pygame.sprite.RenderUpdates"><code class="xref py py-class docutils literal notranslate"><span class="pre">RenderUpdates</span></code></a></p>
282  <blockquote>
283  <div><p>This is the Cadillac of rendering <code class="docutils literal notranslate"><span class="pre">Groups</span></code>. It is inherited from
284  <code class="docutils literal notranslate"><span class="pre">RenderClear</span></code>, but changes the <code class="docutils literal notranslate"><span class="pre">draw()</span></code> method to also return a list of
285  pygame <code class="docutils literal notranslate"><span class="pre">Rects</span></code>, which represent all the areas on screen that have been
286  changed.</p>
287  </div></blockquote>
288  </div></blockquote>
289  <p>That is the list of different groups available We'll discuss more about these
290  rendering groups in the next section. There's nothing stopping you from
291  creating your own Group classes as well. They are just python code, so you can
292  inherit from one of these and add/change whatever you want. In the future I
293  hope we can add a couple more <code class="docutils literal notranslate"><span class="pre">Groups</span></code> to this list. A <code class="docutils literal notranslate"><span class="pre">GroupMulti</span></code> which
294  is like the <code class="docutils literal notranslate"><span class="pre">GroupSingle</span></code>, but can hold up to a given number of sprites (in
295  some sort of circular buffer?). Also a super-render group that can clear the
296  position of the old sprites without needing a background image to do it (by
297  grabbing a copy of the screen before blitting). Who knows really, but in the
298  future we can add more useful classes to this list.</p>
299  </section>
300  <section id="the-rendering-groups">
301  <h3>The Rendering Groups<a class="headerlink" href="#the-rendering-groups" title="Permalink to this headline">¶</a></h3>
302  <p>From above we can see there are three different rendering groups. We could
303  probably just get away with the <code class="docutils literal notranslate"><span class="pre">RenderUpdates</span></code> one, but it adds overhead not
304  really needed for something like a scrolling game. So we have a couple tools
305  here, pick the right one for the right job.</p>
306  <p>For a scrolling type game, where the background completely changes every frame,
307  we obviously don't need to worry about python's update rectangles in the call
308  to <code class="docutils literal notranslate"><span class="pre">display.update()</span></code>. You should definitely go with the <code class="docutils literal notranslate"><span class="pre">RenderPlain</span></code>
309  group here to manage your rendering.</p>
310  <p>For games where the background is more stationary, you definitely don't want
311  pygame updating the entire screen (since it doesn't need to). This type of game
312  usually involves erasing the old position of each object, then drawing it in a
313  new place for each frame. This way we are only changing what is necessary.
314  Most of the time you will just want to use the <code class="docutils literal notranslate"><span class="pre">RenderUpdates</span></code> class here.
315  Since you will also want to pass this list of changes to the
316  <code class="docutils literal notranslate"><span class="pre">display.update()</span></code> function.</p>
317  <p>The <code class="docutils literal notranslate"><span class="pre">RenderUpdates</span></code> class also does a good job at minimizing overlapping
318  areas in the list of updated rectangles. If the previous position and current
319  position of an object overlap, it will merge them into a single rectangle.
320  Combined with the fact that it properly handles deleted objects, this is
321  one powerful <code class="docutils literal notranslate"><span class="pre">Group</span></code> class. If you've written a game that manages the changed
322  rectangles for the objects in a game, you know this the cause for a lot of
323  messy code in your game. Especially once you start to throw in objects that can
324  be deleted at any time. All this work is reduced to a <code class="docutils literal notranslate"><span class="pre">clear()</span></code> and
325  <code class="docutils literal notranslate"><span class="pre">draw()</span></code> method with this monster class. Plus with the overlap checking, it
326  is likely faster than when you did it manually.</p>
327  <p>Also note that there's nothing stopping you from mixing and matching these
328  render groups in your game. You should definitely use multiple rendering groups
329  when you want to do layering with your sprites. Also if the screen is split
330  into multiple sections, perhaps each section of the screen should use an
331  appropriate render group?</p>
332  </section>
333  <section id="collision-detection">
334  <h3>Collision Detection<a class="headerlink" href="#collision-detection" title="Permalink to this headline">¶</a></h3>
335  <p>The sprite module also comes with two very generic collision detection
336  functions.  For more complex games, these really won't work for you, but you
337  can easily grab the source code for them, and modify them as needed.</p>
338  <p>Here's a summary of what they are, and what they do.</p>
339  <blockquote>
340  <div><p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.spritecollide" title="pygame.sprite.spritecollide"><code class="xref py py-func docutils literal notranslate"><span class="pre">spritecollide(sprite,</span> <span class="pre">group,</span> <span class="pre">dokill)</span> <span class="pre">-&gt;</span> <span class="pre">list</span></code></a></p>
341  <blockquote>
342  <div><p>This checks for collisions between a single sprite and the sprites in a group.
343  It requires a &quot;rect&quot; attribute for all the sprites used. It returns a list of
344  all the sprites that overlap with the first sprite. The &quot;dokill&quot; argument is a
345  boolean argument. If it is true, the function will call the <code class="docutils literal notranslate"><span class="pre">kill()</span></code> method
346  on all the sprites. This means the last reference to each sprite is probably in
347  the returned list. Once the list goes away so do the sprites.  A quick example
348  of using this in a loop</p>
349  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="gp">&gt;&gt;&gt; </span><span class="k">for</span> <span class="n">bomb</span> <span class="ow">in</span> <span class="n">sprite</span><span class="o">.</span><span class="n">spritecollide</span><span class="p">(</span><span class="n">player</span><span class="p">,</span> <span class="n">bombs</span><span class="p">,</span> <span class="mi">1</span><span class="p">):</span>
350  <span class="gp">... </span>    <span class="n">boom_sound</span><span class="o">.</span><span class="n">play</span><span class="p">()</span>
351  <span class="gp">... </span>    <span class="n">Explosion</span><span class="p">(</span><span class="n">bomb</span><span class="p">,</span> <span class="mi">0</span><span class="p">)</span>
352  </pre></div>
353  </div>
354  <p>This finds all the sprites in the &quot;bomb&quot; group that collide with the player.
355  Because of the &quot;dokill&quot; argument it deletes all the crashed bombs. For each
356  bomb that did collide, it plays a &quot;boom&quot; sound effect, and creates a new
357  <code class="docutils literal notranslate"><span class="pre">Explosion</span></code> where the bomb was. (Note, the <code class="docutils literal notranslate"><span class="pre">Explosion</span></code> class here knows to
358  add each instance to the appropriate class, so we don't need to store it in a
359  variable, that last line might feel a little &quot;funny&quot; to you python programmers.</p>
360  </div></blockquote>
361  <p><a class="reference internal" href="../ref/sprite.html#pygame.sprite.groupcollide" title="pygame.sprite.groupcollide"><code class="xref py py-func docutils literal notranslate"><span class="pre">groupcollide(group1,</span> <span class="pre">group2,</span> <span class="pre">dokill1,</span> <span class="pre">dokill2)</span> <span class="pre">-&gt;</span> <span class="pre">dictionary</span></code></a></p>
362  <blockquote>
363  <div><p>This is similar to the <code class="docutils literal notranslate"><span class="pre">spritecollide</span></code> function, but a little more complex.
364  It checks for collisions for all the sprites in one group, to the sprites in
365  another. There is a <code class="docutils literal notranslate"><span class="pre">dokill</span></code> argument for the sprites in each list. When
366  <code class="docutils literal notranslate"><span class="pre">dokill1</span></code> is true, the colliding sprites in <code class="docutils literal notranslate"><span class="pre">group1</span></code> will be <code class="docutils literal notranslate"><span class="pre">kill()``ed.</span>
367  <span class="pre">When</span> <span class="pre">``dokill2</span></code> is true, we get the same results for <code class="docutils literal notranslate"><span class="pre">group2</span></code>. The
368  dictionary it returns works like this; each key in the dictionary is a sprite
369  from <code class="docutils literal notranslate"><span class="pre">group1</span></code> that had a collision.  The value for that key is a list of the
370  sprites that it collided with. Perhaps another quick code sample explains it
371  best</p>
372  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="gp">&gt;&gt;&gt; </span><span class="k">for</span> <span class="n">alien</span> <span class="ow">in</span> <span class="n">sprite</span><span class="o">.</span><span class="n">groupcollide</span><span class="p">(</span><span class="n">aliens</span><span class="p">,</span> <span class="n">shots</span><span class="p">,</span> <span class="mi">1</span><span class="p">,</span> <span class="mi">1</span><span class="p">)</span><span class="o">.</span><span class="n">keys</span><span class="p">()</span>
373  <span class="gp">... </span>    <span class="n">boom_sound</span><span class="o">.</span><span class="n">play</span><span class="p">()</span>
374  <span class="gp">... </span>    <span class="n">Explosion</span><span class="p">(</span><span class="n">alien</span><span class="p">,</span> <span class="mi">0</span><span class="p">)</span>
375  <span class="gp">... </span>    <span class="n">kills</span> <span class="o">+=</span> <span class="mi">1</span>
376  </pre></div>
377  </div>
378  <p>This code checks for the collisions between player bullets and all the aliens
379  they might intersect. In this case we only loop over the dictionary keys, but
380  we could loop over the <code class="docutils literal notranslate"><span class="pre">values()</span></code> or <code class="docutils literal notranslate"><span class="pre">items()</span></code> if we wanted to do something
381  to the specific shots that collided with aliens. If we did loop over the
382  <code class="docutils literal notranslate"><span class="pre">values()</span></code> we would be looping through lists that contain sprites. The same
383  sprite may even appear more than once in these different loops, since the same
384  &quot;shot&quot; could have collided against multiple &quot;aliens&quot;.</p>
385  </div></blockquote>
386  </div></blockquote>
387  <p>Those are the basic collision functions that come with pygame. It should be
388  easy to roll your own that perhaps use something different than the &quot;rect&quot;
389  attribute. Or maybe try to fine-tweak your code a little more by directly
390  effecting the collision object, instead of building a list of the collision?
391  The code in the sprite collision functions is very optimized, but you could
392  speed it up slightly by taking out some functionality you don't need.</p>
393  </section>
394  <section id="common-problems">
395  <h3>Common Problems<a class="headerlink" href="#common-problems" title="Permalink to this headline">¶</a></h3>
396  <p>Currently there is one main problem that catches new users. When you derive
397  your new sprite class with the Sprite base, you <strong>must</strong> call the
398  <code class="docutils literal notranslate"><span class="pre">Sprite.__init__()</span></code> method from your own class <code class="docutils literal notranslate"><span class="pre">__init__()</span></code> method.  If you
399  forget to call the <code class="docutils literal notranslate"><span class="pre">Sprite.__init__()</span></code> method, you get a cryptic error, like
400  this</p>
401  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="ne">AttributeError</span><span class="p">:</span> <span class="s1">&#39;mysprite&#39;</span> <span class="n">instance</span> <span class="n">has</span> <span class="n">no</span> <span class="n">attribute</span> <span class="s1">&#39;_Sprite__g&#39;</span>
402  </pre></div>
403  </div>
404  </section>
405  <section id="extending-your-own-classes-advanced">
406  <h3>Extending Your Own Classes <em>(Advanced)</em><a class="headerlink" href="#extending-your-own-classes-advanced" title="Permalink to this headline">¶</a></h3>
407  <p>Because of speed concerns, the current <code class="docutils literal notranslate"><span class="pre">Group</span></code> classes try to only do exactly
408  what they need, and not handle a lot of general situations. If you decide you
409  need extra features, you may want to create your own <code class="docutils literal notranslate"><span class="pre">Group</span></code> class.</p>
410  <p>The <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> and <code class="docutils literal notranslate"><span class="pre">Group</span></code> classes were designed to be extended, so feel free
411  to create your own <code class="docutils literal notranslate"><span class="pre">Group</span></code> classes to do specialized things. The best place
412  to start is probably the actual python source code for the sprite module.
413  Looking at the current <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> groups should be enough example on how to
414  create your own.</p>
415  <p>For example, here is the source code for a rendering <code class="docutils literal notranslate"><span class="pre">Group</span></code> that calls a
416  <code class="docutils literal notranslate"><span class="pre">render()</span></code> method for each sprite, instead of just blitting an &quot;image&quot;
417  variable from it.  Since we want it to also handle updated areas, we will start
418  with a copy of the original <code class="docutils literal notranslate"><span class="pre">RenderUpdates</span></code> group, here is the code:</p>
419  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="k">class</span> <span class="nc">RenderUpdatesDraw</span><span class="p">(</span><span class="n">RenderClear</span><span class="p">):</span>
420      <span class="sd">&quot;&quot;&quot;call sprite.draw(screen) to render sprites&quot;&quot;&quot;</span>
421      <span class="k">def</span> <span class="nf">draw</span><span class="p">(</span><span class="bp">self</span><span class="p">,</span> <span class="n">surface</span><span class="p">):</span>
422          <span class="n">dirty</span> <span class="o">=</span> <span class="bp">self</span><span class="o">.</span><span class="n">lostsprites</span>
423          <span class="bp">self</span><span class="o">.</span><span class="n">lostsprites</span> <span class="o">=</span> <span class="p">[]</span>
424          <span class="k">for</span> <span class="n">s</span><span class="p">,</span> <span class="n">r</span> <span class="ow">in</span> <span class="bp">self</span><span class="o">.</span><span class="n">spritedict</span><span class="o">.</span><span class="n">items</span><span class="p">():</span>
425              <span class="n">newrect</span> <span class="o">=</span> <span class="n">s</span><span class="o">.</span><span class="n">draw</span><span class="p">(</span><span class="n">screen</span><span class="p">)</span> <span class="c1">#Here&#39;s the big change</span>
426              <span class="k">if</span> <span class="n">r</span> <span class="ow">is</span> <span class="mi">0</span><span class="p">:</span>
427                  <span class="n">dirty</span><span class="o">.</span><span class="n">append</span><span class="p">(</span><span class="n">newrect</span><span class="p">)</span>
428              <span class="k">else</span><span class="p">:</span>
429                  <span class="n">dirty</span><span class="o">.</span><span class="n">append</span><span class="p">(</span><span class="n">newrect</span><span class="o">.</span><span class="n">union</span><span class="p">(</span><span class="n">r</span><span class="p">))</span>
430              <span class="bp">self</span><span class="o">.</span><span class="n">spritedict</span><span class="p">[</span><span class="n">s</span><span class="p">]</span> <span class="o">=</span> <span class="n">newrect</span>
431          <span class="k">return</span> <span class="n">dirty</span>
432  </pre></div>
433  </div>
434  <p>Following is more information on how you could create your own <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> and
435  <code class="docutils literal notranslate"><span class="pre">Group</span></code> objects from scratch.</p>
436  <p>The <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> objects only &quot;require&quot; two methods. &quot;add_internal()&quot; and
437  &quot;remove_internal()&quot;.  These are called by the <code class="docutils literal notranslate"><span class="pre">Group</span></code> classes when they are
438  removing a sprite from themselves. The <code class="docutils literal notranslate"><span class="pre">add_internal()</span></code> and
439  <code class="docutils literal notranslate"><span class="pre">remove_internal()</span></code> have a single argument which is a group. Your <code class="docutils literal notranslate"><span class="pre">Sprite</span></code>
440  will need some way to also keep track of the <code class="docutils literal notranslate"><span class="pre">Groups</span></code> it belongs to. You will
441  likely want to try to match the other methods and arguments to the real
442  <code class="docutils literal notranslate"><span class="pre">Sprite</span></code> class, but if you're not going to use those methods, you sure don't
443  need them.</p>
444  <p>It is almost the same requirements for creating your own <code class="docutils literal notranslate"><span class="pre">Group</span></code>. In fact, if
445  you look at the source you'll see the <code class="docutils literal notranslate"><span class="pre">GroupSingle</span></code> isn't derived from the
446  <code class="docutils literal notranslate"><span class="pre">Group</span></code> class, it just implements the same methods so you can't really tell
447  the difference. Again you need an &quot;add_internal()&quot; and &quot;remove_internal()&quot;
448  method that the sprites call when they want to belong or remove themselves from
449  the group. The <code class="docutils literal notranslate"><span class="pre">add_internal()</span></code> and <code class="docutils literal notranslate"><span class="pre">remove_internal()</span></code> have a single
450  argument which is a sprite. The only other requirement for the <code class="docutils literal notranslate"><span class="pre">Group</span></code>
451  classes is they have a dummy attribute named &quot;_spritegroup&quot;. It doesn't matter
452  what the value is, as long as the attribute is present. The Sprite classes can
453  look for this attribute to determine the difference between a &quot;group&quot; and any
454  ordinary python container. (This is important, because several sprite methods
455  can take an argument of a single group, or a sequence of groups. Since they
456  both look similar, this is the most flexible way to &quot;see&quot; the difference.)</p>
457  <p>You should go through the code for the sprite module. While the code is a bit
458  &quot;tuned&quot;, it's got enough comments to help you follow along.  There's even a
459  TODO section in the source if you feel like contributing.</p>
