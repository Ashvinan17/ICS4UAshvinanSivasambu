6    .. module:: pygame.midi
7       :synopsis: pygame module for interacting with midi input and output.
8    
9    | :sl:`pygame module for interacting with midi input and output.`
10   
11   .. versionadded:: 1.9.0
12   
13   The midi module can send output to midi devices and get input from midi
14   devices. It can also list midi devices on the system.
15   
16   The midi module supports real and virtual midi devices.
17   
18   It uses the portmidi library. Is portable to which ever platforms portmidi
19   supports (currently Windows, Mac OS X, and Linux).
20   
21   This uses pyportmidi for now, but may use its own bindings at some point in the
22   future. The pyportmidi bindings are included with pygame.
23

129  .. class:: Output
130  
131     | :sl:`Output is used to send midi to an output device`
132     | :sg:`Output(device_id) -> None`
133     | :sg:`Output(device_id, latency=0) -> None`
134     | :sg:`Output(device_id, buffer_size=256) -> None`
135     | :sg:`Output(device_id, latency, buffer_size) -> None`
136  
137     The ``buffer_size`` specifies the number of output events to be buffered
138     waiting for output. In some cases (see below) PortMidi does not buffer
139     output at all and merely passes data to a lower-level API, in which case
140     buffersize is ignored.
141  
142     ``latency`` is the delay in milliseconds applied to timestamps to determine
143     when the output should actually occur. If ``latency`` is <<0, 0 is assumed.
144  
145     If ``latency`` is zero, timestamps are ignored and all output is delivered
146     immediately. If ``latency`` is greater than zero, output is delayed until the
147     message timestamp plus the ``latency``. In some cases, PortMidi can obtain
148     better timing than your application by passing timestamps along to the
149     device driver or hardware. Latency may also help you to synchronize midi
150     data to audio data by matching midi latency to the audio buffer latency.
151  
152     .. note::
153        Time is measured relative to the time source indicated by time_proc.
154        Timestamps are absolute, not relative delays or offsets.
155  
156     .. method:: abort
157  
158        | :sl:`terminates outgoing messages immediately`
159        | :sg:`abort() -> None`
160  
161        The caller should immediately close the output port; this call may result
162        in transmission of a partial midi message. There is no abort for Midi
163        input because the user can simply ignore messages in the buffer and close

208     .. method:: pitch_bend
209  
210        | :sl:`modify the pitch of a channel.`
211        | :sg:`set_instrument(value=0, channel=0) -> None`
212  
213        Adjust the pitch of a channel. The value is a signed integer
214        from -8192 to +8191. For example, 0 means "no change", +4096 is
215        typically a semitone higher, and -8192 is 1 whole tone lower (though
216        the musical range corresponding to the pitch bend range can also be
217        changed in some synthesizers).
218  
219        If no value is given, the pitch bend is returned to "no change".
220

223     .. method:: write
224  
225        | :sl:`writes a list of midi data to the Output`
226        | :sg:`write(data) -> None`
227  
228        Writes series of MIDI information in the form of a list.
229  
230        :param list data: data to write, the expected format is
231           ``[[[status, data1=0, data2=0, ...], timestamp], ...]``
232           with the ``data#`` fields being optional
233  
234        :raises IndexError: if more than 1024 elements in the data list
235  
236        Example:
237        ::
238  
239           # Program change at time 20000 and 500ms later send note 65 with
240           # velocity 100.
241           write([[[0xc0, 0, 0], 20000], [[0x90, 60, 100], 20500]])
242  
243        .. note::
244           - Timestamps will be ignored if latency = 0
245           - To get a note to play immediately, send MIDI info with timestamp
246             read from function Time
247           - Optional data fields: ``write([[[0xc0, 0, 0], 20000]])`` is
248             equivalent to ``write([[[0xc0], 20000]])``
249

310  .. function:: get_default_input_id
311  
312     | :sl:`gets default input device number`
313     | :sg:`get_default_input_id() -> default_id`
314  
315     The following describes the usage details for this function and the
316     :func:`get_default_output_id` function.
317  
318     Return the default device ID or ``-1`` if there are no devices. The result
319     can be passed to the :class:`Input`/:class:`Output` class.
320  
321     On a PC the user can specify a default device by setting an environment
322     variable. To use device #1, for example:
323     ::
324  
325        set PM_RECOMMENDED_INPUT_DEVICE=1
326        or
327        set PM_RECOMMENDED_OUTPUT_DEVICE=1
328  
329     The user should first determine the available device ID by using the
330     supplied application "testin" or "testout".
331  
332     In general, the registry is a better place for this kind of info. With
333     USB devices that can come and go, using integers is not very reliable
334     for device identification. Under Windows, if ``PM_RECOMMENDED_INPUT_DEVICE``
335     (or ``PM_RECOMMENDED_OUTPUT_DEVICE``) is NOT found in the environment,
336     then the default device is obtained by looking for a string in the registry
337     under:
338     ::
339  
340        HKEY_LOCAL_MACHINE/SOFTWARE/PortMidi/Recommended_Input_Device
341        or
342        HKEY_LOCAL_MACHINE/SOFTWARE/PortMidi/Recommended_Output_Device
343  
344  
345     The number of the first device with a substring that matches the
346     string exactly is returned. For example, if the string in the registry is
347     "USB" and device 1 is named "In USB MidiSport 1x1", then that will be
348     the default input because it contains the string "USB".
349  
350     In addition to the name, :func:`get_device_info()` returns "interf", which is
351     the interface name. The "interface" is the underlying software system or
352     API used by PortMidi to access devices. Supported interfaces:
353     ::
354  
355        MMSystem   # the only Win32 interface currently supported
356        ALSA       # the only Linux interface currently supported
357        CoreMIDI   # the only Mac OS X interface currently supported
358        # DirectX - not implemented
359        # OSS     - not implemented
360  
361     To specify both the interface and the device name in the registry, separate
362     the two with a comma and a space. The string before the comma must be a
363     substring of the "interf" string and the string after the space must be a
364     substring of the "name" name string in order to match the device. e.g.:
365     ::
366  
367        MMSystem, In USB MidiSport 1x1
368  
369     .. note::
370        In the current release, the default is simply the first device (the
371        input or output device with the lowest PmDeviceID).

384  .. function:: get_device_info
385  
386     | :sl:`returns information about a midi device`
387     | :sg:`get_device_info(an_id) -> (interf, name, input, output, opened)`
388     | :sg:`get_device_info(an_id) -> None`
389  
390     Gets the device info for a given id.
391  
392     :param int an_id: id of the midi device being queried
393  
394     :returns: if the id is out of range ``None`` is returned, otherwise
395        a tuple of (interf, name, input, output, opened) is returned.
396  
397           - interf: string describing the device interface (e.g. 'ALSA')
398           - name: string name of the device (e.g. 'Midi Through Port-0')
399           - input: 1 if the device is an input device, otherwise 0
400           - output: 1 if the device is an output device, otherwise 0
401           - opened: 1 if the device is opened, otherwise 0
402     :rtype: tuple or None
