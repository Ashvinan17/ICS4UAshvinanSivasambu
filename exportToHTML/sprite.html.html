205  <p>This module contains several simple classes to be used within games. There is
206  the main Sprite class and several Group classes that contain Sprites. The use
207  of these classes is entirely optional when using pygame. The classes are fairly
208  lightweight and only provide a starting place for the code that is common to
209  most games.</p>
210  <p>The Sprite class is intended to be used as a base class for the different types
211  of objects in the game. There is also a base Group class that simply stores
212  sprites. A game could create new types of Group classes that operate on
213  specially customized Sprite instances they contain.</p>
214  <p>The basic Sprite class can draw the Sprites it contains to a Surface. The
215  <code class="docutils literal notranslate"><span class="pre">Group.draw()</span></code> method requires that each Sprite have a <code class="docutils literal notranslate"><span class="pre">Surface.image</span></code>
216  attribute and a <code class="docutils literal notranslate"><span class="pre">Surface.rect</span></code>. The <code class="docutils literal notranslate"><span class="pre">Group.clear()</span></code> method requires these
217  same attributes, and can be used to erase all the Sprites with background.
218  There are also more advanced Groups: <code class="docutils literal notranslate"><span class="pre">pygame.sprite.RenderUpdates()</span></code> and
219  <code class="docutils literal notranslate"><span class="pre">pygame.sprite.OrderedUpdates()</span></code>.</p>
220  <p>Lastly, this module contains several collision functions. These help find
221  sprites inside multiple groups that have intersecting bounding rectangles. To
222  find the collisions, the Sprites are required to have a <code class="docutils literal notranslate"><span class="pre">Surface.rect</span></code>
223  attribute assigned.</p>
224  <p>The groups are designed for high efficiency in removing and adding Sprites to
225  them. They also allow cheap testing to see if a Sprite already exists in a
226  Group. A given Sprite can exist in any number of groups. A game could use some
227  groups to control object rendering, and a completely separate set of groups to
228  control interaction or player movement. Instead of adding type attributes or
229  bools to a derived Sprite class, consider keeping the Sprites inside organized
230  Groups. This will allow for easier lookup later in the game.</p>
231  <p>Sprites and Groups manage their relationships with the <code class="docutils literal notranslate"><span class="pre">add()</span></code> and
232  <code class="docutils literal notranslate"><span class="pre">remove()</span></code> methods. These methods can accept a single or multiple targets for
233  membership. The default initializers for these classes also takes a single or
234  list of targets for initial membership. It is safe to repeatedly add and remove
235  the same Sprite from a Group.</p>
236  <p>While it is possible to design sprite and group classes that don't derive from
237  the Sprite and AbstractGroup classes below, it is strongly recommended that you
238  extend those when you add a Sprite or Group class.</p>
239  <p>Sprites are not thread safe. So lock them yourself if using threads.</p>

784  <p>This group is fully compatible with <a class="tooltip reference internal" href="#pygame.sprite.Sprite" title=""><code class="xref py py-class docutils literal notranslate"><span class="pre">pygame.sprite.Sprite</span></code><span class="tooltip-content">Simple base class for visible game objects.</span></a>.</p>
785  <p>You can set the default layer through kwargs using 'default_layer' and an
786  integer for the layer. The default layer is 0.</p>
787  <p>If the sprite you add has an attribute _layer then that layer will be used.
788  If the **kwarg contains 'layer' then the sprites passed will be added to
789  that layer (overriding the <code class="docutils literal notranslate"><span class="pre">sprite.layer</span></code> attribute). If neither sprite
790  has attribute layer nor **kwarg then the default layer is used to add the
791  sprites.</p>

1308 <div class="line"><span class="signature">groupcollide(group1, group2, dokill1, dokill2, collided = None) -&gt; Sprite_dict</span></div>
1309 </div>
1310 <p>This will find collisions between all the Sprites in two groups.
1311 Collision is determined by comparing the <code class="docutils literal notranslate"><span class="pre">Sprite.rect</span></code> attribute of
1312 each Sprite or by using the collided function if it is not None.</p>
1313 <p>Every Sprite inside group1 is added to the return dictionary. The value for
1314 each item is the list of Sprites in group2 that intersect.</p>
1315 <p>If either dokill argument is True, the colliding Sprites will be removed
1316 from their respective Group.</p>
1317 <p>The collided argument is a callback function used to calculate if two sprites are
1318 colliding. It should take two sprites as values and return a bool value
1319 indicating if they are colliding. If collided is not passed, then all
1320 sprites must have a &quot;rect&quot; value, which is a rectangle of the sprite area,
1321 which will be used to calculate the collision.</p>
