101  <span id="id1"></span><h2>2. Revision: Pygame fundamentals<a class="headerlink" href="#makegames-2" title="Permalink to this headline">¶</a></h2>
102  <section id="the-basic-pygame-game">
103  <span id="makegames-2-1"></span><h3>2.1. The basic Pygame game<a class="headerlink" href="#the-basic-pygame-game" title="Permalink to this headline">¶</a></h3>
104  <p>For the sake of revision, and to ensure that you are familiar with the basic structure of a Pygame program, I'll briefly run through
105  a basic Pygame program, which will display no more than a window with some text in it, that should, by the end, look something like
106  this (though of course the window decoration will probably be different on your system):</p>
107  <img alt="../_images/tom_basic.png" src="../_images/tom_basic.png" />
108  <p>The full code for this example looks like this:</p>
109  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="ch">#!/usr/bin/python</span>
110  
111  <span class="kn">import</span> <span class="nn">pygame</span>

150  <section id="basic-pygame-objects">
151  <span id="makegames-2-2"></span><h3>2.2. Basic Pygame objects<a class="headerlink" href="#basic-pygame-objects" title="Permalink to this headline">¶</a></h3>
152  <p>As you can see, the code consists of three main objects: the screen, the background, and the text. Each of these objects is created
153  by first calling an instance of an in-built Pygame object, and then modifying it to fit our needs. The screen is a slightly special
154  case, because we still modify the display through Pygame calls, rather than calling methods belonging to the screen object. But for
155  all other Pygame objects, we first create the object as a copy of a Pygame object, giving it some attributes, and build our game
156  objects from them.</p>
157  <p>With the background, we first create a Pygame Surface object, and make it the size of the screen. We then perform the convert()
158  operation to convert the Surface to a single pixel format. This is more obviously necessary when we have several images and surfaces,
159  all of different pixel formats, which makes rendering them quite slow. By converting all the surfaces, we can drastically speed up
160  rendering times. Finally, we fill the background surface with white (255, 255, 255). These values are <em class="firstterm">RGB</em> (Red Green
161  Blue), and can be worked out from any good paint program.</p>
162  <p>With the text, we require more than one object. First, we create a font object, which defines which font to use, and the size of the
163  font. Then we create a text object, by using the <code class="docutils literal notranslate"><span class="pre">render</span></code> method that belongs to our font object, supplying three arguments:
164  the text to be rendered, whether or not it should be anti-aliased (1=yes, 0=no), and the color of the text (again in RGB format). Next
165  we create a third text object, which gets the rectangle for the text. The easiest way to understand this is to imagine drawing a
166  rectangle that will surround all of the text; you can then use this rectangle to get/set the position of the text on the screen. So
167  in this example we get the rectangle, set its <code class="docutils literal notranslate"><span class="pre">centerx</span></code> attribute to be the <code class="docutils literal notranslate"><span class="pre">centerx</span></code> attribute of the
168  background (so the text's center will be the same as the background's center, i.e. the text will be centered on the screen on the x
169  axis). We could also set the y coordinate, but it's not any different so I left the text at the top of the screen. As the screen is
170  small anyway, it didn't seem necessary.</p>
171  </section>
172  <section id="blitting">
173  <span id="makegames-2-3"></span><h3>2.3. Blitting<a class="headerlink" href="#blitting" title="Permalink to this headline">¶</a></h3>
174  <p>Now we have created our game objects, we need to actually render them. If we didn't and we ran the program, we'd just see a
175  blank window, and the objects would remain invisible. The term used for rendering objects is <em class="firstterm">blitting</em>, which is where
176  you copy the pixels belonging to said object onto the destination object. So to render the background object, you blit it onto the
177  screen. In this example, to make things simple, we blit the text onto the background (so the background will now have a copy of the
178  text on it), and then blit the background onto the screen.</p>
179  <p>Blitting is one of the slowest operations in any game, so you need to be careful not to blit too much onto the screen in every frame.
180  If you have a background image, and a ball flying around the screen, then you could blit the background and then the ball in every
181  frame, which would cover up the ball's previous position and render the new ball, but this would be pretty slow. A better solution is
182  to blit the background onto the area that the ball previously occupied, which can be found by the ball's previous rectangle, and then
183  blitting the ball, so that you are only blitting two small areas.</p>
184  </section>
185  <section id="the-event-loop">
186  <span id="makegames-2-4"></span><h3>2.4. The event loop<a class="headerlink" href="#the-event-loop" title="Permalink to this headline">¶</a></h3>
187  <p>Once you've set the game up, you need to put it into a loop so that it will continuously run until the user signals that he/she wants
188  to exit. So you start an open <code class="docutils literal notranslate"><span class="pre">while</span></code> loop, and then for each iteration of the loop, which will be each frame of the game,
189  update the game. The first thing is to check for any Pygame events, which will be the user hitting the keyboard, clicking a mouse
190  button, moving a joystick, resizing the window, or trying to close it. In this case, we simply want to watch out for for user trying
191  to quit the game by closing the window, in which case the game should <code class="docutils literal notranslate"><span class="pre">return</span></code>, which will end the <code class="docutils literal notranslate"><span class="pre">while</span></code> loop.
192  Then we simply need to re-blit the background, and flip (update) the display to have everything drawn. OK, as nothing moves or happens
193  in this example, we don't strictly speaking need to re-blit the background in every iteration, but I put it in because when things are
194  moving around on the screen, you will need to do all your blitting here.</p>
195  </section>
196  <section id="ta-da">
197  <span id="makegames-2-5"></span><h3>2.5. Ta-da!<a class="headerlink" href="#ta-da" title="Permalink to this headline">¶</a></h3>
198  <p>And that's it - your most basic Pygame game! All games will take a form similar to this, but with lots more code for the actual game
199  functions themselves, which are more to do your with programming, and less guided in structure by the workings of Pygame. This is what
200  this tutorial is really about, and will now go onto.</p>
201  </section>
