6    .. module:: pygame.display
7       :synopsis: pygame module to control the display window and screen
8    
9    | :sl:`pygame module to control the display window and screen`
10   
11   This module offers control over the pygame display. Pygame has a single display
12   Surface that is either contained in a window or runs full screen. Once you
13   create the display you treat it as a regular Surface. Changes are not
14   immediately visible onscreen; you must choose one of the two flipping functions
15   to update the actual display.
16   
17   The origin of the display, where x = 0 and y = 0, is the top left of the
18   screen. Both axes increase positively towards the bottom right of the screen.
19   
20   The pygame display can actually be initialized in one of several modes. By
21   default, the display is a basic software driven framebuffer. You can request
22   special modules like automatic scaling or OpenGL support. These are
23   controlled by flags passed to ``pygame.display.set_mode()``.
24   
25   Pygame can only have a single display active at any time. Creating a new one
26   with ``pygame.display.set_mode()`` will close the previous display. To detect
27   the number and size of attached screens, you can use
28   ``pygame.display.get_desktop_sizes`` and then select appropriate window size
29   and display index to pass to ``pygame.display.set_mode()``.
30   
31   For backward compatibility ``pygame.display`` allows precise control over
32   the pixel format or display resolutions. This used to be necessary with old
33   grahics cards and CRT screens, but is usually not needed any more. Use the
34   functions ``pygame.display.mode_ok()``, ``pygame.display.list_modes()``, and
35   ``pygame.display.Info()`` to query detailed information about the display.
36   
37   Once the display Surface is created, the functions from this module affect the
38   single existing display. The Surface becomes invalid if the module is
39   uninitialized. If a new display mode is set, the existing Surface will
40   automatically switch to operate on the new display.
41   
42   When the display mode is set, several events are placed on the pygame event
43   queue. ``pygame.QUIT`` is sent when the user has requested the program to
44   shut down. The window will receive ``pygame.ACTIVEEVENT`` events as the display
45   gains and loses input focus. If the display is set with the
46   ``pygame.RESIZABLE`` flag, ``pygame.VIDEORESIZE`` events will be sent when the
47   user adjusts the window dimensions. Hardware displays that draw direct to the
48   screen will get ``pygame.VIDEOEXPOSE`` events when portions of the window must
49   be redrawn.
50   
51   A new windowevent API was introduced in pygame 2.0.1. Check event module docs
52   for more information on that
53   
54   Some display environments have an option for automatically stretching all
55   windows. When this option is enabled, this automatic stretching distorts the
56   appearance of the pygame window. In the pygame examples directory, there is
57   example code (prevent_display_stretching.py) which shows how to disable this
58   automatic stretching of the pygame display on Microsoft Windows (Vista or newer
59   required).

61   .. function:: init
62   
63      | :sl:`Initialize the display module`
64      | :sg:`init() -> None`
65   
66      Initializes the pygame display module. The display module cannot do anything
67      until it is initialized. This is usually handled for you automatically when
68      you call the higher level ``pygame.init()``.
69   
70      Pygame will select from one of several internal display backends when it is
71      initialized. The display mode will be chosen depending on the platform and
72      permissions of current user. Before the display module is initialized the
73      environment variable ``SDL_VIDEODRIVER`` can be set to control which backend
74      is used. The systems with multiple choices are listed here.
75   
76      ::
77   
78         Windows : windib, directx
79         Unix    : x11, dga, fbcon, directfb, ggi, vgl, svgalib, aalib
80   
81      On some platforms it is possible to embed the pygame display into an already
82      existing window. To do this, the environment variable ``SDL_WINDOWID`` must
83      be set to a string containing the window id or handle. The environment
84      variable is checked when the pygame display is initialized. Be aware that
85      there can be many strange side effects when running in an embedded display.
86   
87      It is harmless to call this more than once, repeated calls have no effect.

113  .. function:: set_mode
114  
115     | :sl:`Initialize a window or screen for display`
116     | :sg:`set_mode(size=(0, 0), flags=0, depth=0, display=0, vsync=0) -> Surface`
117  
118     This function will create a display Surface. The arguments passed in are
119     requests for a display type. The actual created display will be the best
120     possible match supported by the system.
121  
122     Note that calling this function implicitly initializes ``pygame.display``, if
123     it was not initialized before.
124  
125     The size argument is a pair of numbers representing the width and
126     height. The flags argument is a collection of additional options. The depth
127     argument represents the number of bits to use for color.
128  
129     The Surface that gets returned can be drawn to like a regular Surface but
130     changes will eventually be seen on the monitor.
131  
132     If no size is passed or is set to ``(0, 0)`` and pygame uses ``SDL``
133     version 1.2.10 or above, the created Surface will have the same size as the
134     current screen resolution. If only the width or height are set to ``0``, the
135     Surface will have the same width or height as the screen resolution. Using a
136     ``SDL`` version prior to 1.2.10 will raise an exception.
137  
138     It is usually best to not pass the depth argument. It will default to the
139     best and fastest color depth for the system. If your game requires a
140     specific color format you can control the depth with this argument. Pygame
141     will emulate an unavailable color depth which can be slow.
142  
143     When requesting fullscreen display modes, sometimes an exact match for the
144     requested size cannot be made. In these situations pygame will select
145     the closest compatible match. The returned surface will still always match
146     the requested size.
147  
148     On high resolution displays(4k, 1080p) and tiny graphics games (640x480)
149     show up very small so that they are unplayable. SCALED scales up the window
150     for you. The game thinks it's a 640x480 window, but really it can be bigger.
151     Mouse events are scaled for you, so your game doesn't need to do it. Note
152     that SCALED is considered an experimental API and may change in future
153     releases.
154  
155     The flags argument controls which type of display you want. There are
156     several to choose from, and you can even combine multiple types using the
157     bitwise or operator, (the pipe "|" character). Here are the display
158     flags you will want to choose from:
159  
160     ::
161  
162        pygame.FULLSCREEN    create a fullscreen display
163        pygame.DOUBLEBUF     (obsolete in pygame 2) recommended for HWSURFACE or OPENGL
164        pygame.HWSURFACE     (obsolete in pygame 2) hardware accelerated, only in FULLSCREEN
165        pygame.OPENGL        create an OpenGL-renderable display
166        pygame.RESIZABLE     display window should be sizeable
167        pygame.NOFRAME       display window will have no border or controls
168        pygame.SCALED        resolution depends on desktop size and scale graphics
169        pygame.SHOWN         window is opened in visible mode (default)
170        pygame.HIDDEN        window is opened in hidden mode
171  
172  
173     .. versionadded:: 2.0.0 ``SCALED``, ``SHOWN`` and ``HIDDEN``
174  
175     By setting the ``vsync`` parameter to ``1``, it is possible to get a display
176     with vertical sync, but you are not guaranteed to get one. The request only
177     works at all for calls to ``set_mode()`` with the ``pygame.OPENGL`` or
178     ``pygame.SCALED`` flags set, and is still not guaranteed even with one of
179     those set. What you get depends on the hardware and driver configuration
180     of the system pygame is running on. Here is an example usage of a call
181     to ``set_mode()`` that may give you a display with vsync:
182  
183     ::
184  
185       flags = pygame.OPENGL | pygame.FULLSCREEN
186       window_surface = pygame.display.set_mode((1920, 1080), flags, vsync=1)
187  
188     Vsync behaviour is considered experimental, and may change in future releases.
189  
190     .. versionadded:: 2.0.0 ``vsync``
191  
192     Basic example:
193  
194     ::
195  
196          # Open a window on the screen
197          screen_width=700
198          screen_height=400
199          screen=pygame.display.set_mode([screen_width, screen_height])
200  
201     The display index ``0`` means the default display is used. If no display
202     index argument is provided, the default display can be overridden with an
203     environment variable.

221  
222     | :sl:`Update the full display Surface to the screen`
223     | :sg:`flip() -> None`
224  
225     This will update the contents of the entire display. If your display mode is
226     using the flags ``pygame.HWSURFACE`` and ``pygame.DOUBLEBUF`` on pygame 1,
227     this will wait for a vertical retrace and swap the surfaces.
228  
229     When using an ``pygame.OPENGL`` display mode this will perform a gl buffer
230     swap.

234  .. function:: update
235  
236     | :sl:`Update portions of the screen for software displays`
237     | :sg:`update(rectangle=None) -> None`
238     | :sg:`update(rectangle_list) -> None`
239  
240     This function is like an optimized version of ``pygame.display.flip()`` for
241     software displays. It allows only a portion of the screen to updated,
242     instead of the entire area. If no argument is passed it updates the entire
243     Surface area like ``pygame.display.flip()``.
244  
245     Note that calling ``display.update(None)`` means no part of the window is
246     updated. Whereas ``display.update()`` means the whole window is updated.
247  
248     You can pass the function a single rectangle, or a sequence of rectangles.
249     It is more efficient to pass many rectangles at once than to call update
250     multiple times with single or a partial list of rectangles. If passing a
251     sequence of rectangles it is safe to include None values in the list, which
252     will be skipped.
253  
254     This call cannot be used on ``pygame.OPENGL`` displays and will generate an
255     exception.

272  .. function:: Info
273  
274     | :sl:`Create a video display information object`
275     | :sg:`Info() -> VideoInfo`
276  
277     Creates a simple object containing several attributes to describe the
278     current graphics environment. If this is called before
279     ``pygame.display.set_mode()`` some platforms can provide information about
280     the default display mode. This can also be called after setting the display
281     mode to verify specific display options were satisfied. The VidInfo object
282     has several attributes:
283  
284     ::
285  
286       hw:         1 if the display is hardware accelerated
287       wm:         1 if windowed display modes can be used
288       video_mem:  The megabytes of video memory on the display. This is 0 if
289                   unknown
290       bitsize:    Number of bits used to store each pixel
291       bytesize:   Number of bytes used to store each pixel
292       masks:      Four values used to pack RGBA values into pixels
293       shifts:     Four values used to pack RGBA values into pixels
294       losses:     Four values used to pack RGBA values into pixels
295       blit_hw:    1 if hardware Surface blitting is accelerated
296       blit_hw_CC: 1 if hardware Surface colorkey blitting is accelerated
297       blit_hw_A:  1 if hardware Surface pixel alpha blitting is accelerated
298       blit_sw:    1 if software Surface blitting is accelerated
299       blit_sw_CC: 1 if software Surface colorkey blitting is accelerated
300       blit_sw_A:  1 if software Surface pixel alpha blitting is accelerated
301       current_h, current_w:  Height and width of the current video mode, or
302                   of the desktop mode if called before the display.set_mode
303                   is called. (current_h, current_w are available since
304                   SDL 1.2.10, and pygame 1.8.0). They are -1 on error, or if
305                   an old SDL is being used.

323  .. function:: get_desktop_sizes
324  
325     | :sl:`Get sizes of active desktops`
326     | :sg:`get_desktop_sizes() -> list`
327  
328     This function returns the sizes of the currrently configured
329     virtual desktops as a list of (x, y) tuples of integers.
330  
331     The length of the list is not the same as the number of attached monitors,
332     as a desktop can be mirrored across multiple monitors. The desktop sizes
333     do not indicate the maximum monitor resolutions supported by the hardware,
334     but the desktop size configured in the operating system.
335  
336     In order to fit windows into the desktop as it is currently configured, and
337     to respect the resolution configured by the operating system in fullscreen
338     mode, this function *should* be used to replace many use cases of
339     ``pygame.display.list_modes()`` whenever applicable.
340  
341     .. versionadded:: 2.0.0
342  
343  .. function:: list_modes
344  
345     | :sl:`Get list of available fullscreen modes`
346     | :sg:`list_modes(depth=0, flags=pygame.FULLSCREEN, display=0) -> list`
347  
348     This function returns a list of possible sizes for a specified color
349     depth. The return value will be an empty list if no display modes are
350     available with the given arguments. A return value of ``-1`` means that
351     any requested size should work (this is likely the case for windowed
352     modes). Mode sizes are sorted from biggest to smallest.
353  
354     If depth is ``0``, the current/best color depth for the display is used.
355     The flags defaults to ``pygame.FULLSCREEN``, but you may need to add
356     additional flags for specific fullscreen modes.
357  
358     The display index ``0`` means the default display is used.
359  
360     Since pygame 2.0, ``pygame.display.get_desktop_sizes()`` has taken over
361     some use cases from ``pygame.display.list_modes()``:
362  
363     To find a suitable size for non-fullscreen windows, it is preferable to
364     use ``pygame.display.get_desktop_sizes()`` to get the size of the *current*
365     desktop, and to then choose a smaller window size. This way, the window is
366     guaranteed to fit, even when the monitor is configured to a lower resolution
367     than the maximum supported by the hardware.
368  
369     To avoid changing the physical monitor resolution, it is also preferable to
370     use ``pygame.display.get_desktop_sizes()`` to determine the fullscreen
371     resolution. Developers are strongly advised to default to the current
372     physical monitor resolution unless the user explicitly requests a different
373     one (e.g. in an options menu or configuration file).
374

379  .. function:: mode_ok
380  
381     | :sl:`Pick the best color depth for a display mode`
382     | :sg:`mode_ok(size, flags=0, depth=0, display=0) -> depth`
383  
384     This function uses the same arguments as ``pygame.display.set_mode()``. It
385     is used to determine if a requested display mode is available. It will
386     return ``0`` if the display mode cannot be set. Otherwise it will return a
387     pixel depth that best matches the display asked for.
388  
389     Usually the depth argument is not passed, but some platforms can support
390     multiple display depths. If passed it will hint to which depth is a better
391     match.
392  
393     The function will return ``0`` if the passed display flags cannot be set.
394  
395     The display index ``0`` means the default display is used.

412  .. function:: gl_set_attribute
413  
414     | :sl:`Request an OpenGL display attribute for the display mode`
415     | :sg:`gl_set_attribute(flag, value) -> None`
416  
417     When calling ``pygame.display.set_mode()`` with the ``pygame.OPENGL`` flag,
418     Pygame automatically handles setting the OpenGL attributes like color and
419     double-buffering. OpenGL offers several other attributes you may want control
420     over. Pass one of these attributes as the flag, and its appropriate value.
421     This must be called before ``pygame.display.set_mode()``.
422  
423     Many settings are the requested minimum. Creating a window with an OpenGL context
424     will fail if OpenGL cannot provide the requested attribute, but it may for example
425     give you a stencil buffer even if you request none, or it may give you a larger
426     one than requested.
427  
428     The ``OPENGL`` flags are:
429  
430     ::
431  
432       GL_ALPHA_SIZE, GL_DEPTH_SIZE, GL_STENCIL_SIZE, GL_ACCUM_RED_SIZE,
433       GL_ACCUM_GREEN_SIZE,  GL_ACCUM_BLUE_SIZE, GL_ACCUM_ALPHA_SIZE,
434       GL_MULTISAMPLEBUFFERS, GL_MULTISAMPLESAMPLES, GL_STEREO
435  
436     :const:`GL_MULTISAMPLEBUFFERS`
437  
438       Whether to enable multisampling anti-aliasing.
439       Defaults to 0 (disabled).
440  
441       Set ``GL_MULTISAMPLESAMPLES`` to a value
442       above 0 to control the amount of anti-aliasing.
443       A typical value is 2 or 3.
444  
445     :const:`GL_STENCIL_SIZE`
446  
447       Minimum bit size of the stencil buffer. Defaults to 0.
448  
449     :const:`GL_DEPTH_SIZE`
450  
451       Minimum bit size of the depth buffer. Defaults to 16.
452  
453     :const:`GL_STEREO`
454  
455       1 enables stereo 3D. Defaults to 0.
456  
457     :const:`GL_BUFFER_SIZE`
458  
459       Minimum bit size of the frame buffer. Defaults to 0.
460  
461     .. versionadded:: 2.0.0 Additional attributes:
462  
463     ::
464  
465       GL_ACCELERATED_VISUAL,
466       GL_CONTEXT_MAJOR_VERSION, GL_CONTEXT_MINOR_VERSION,
467       GL_CONTEXT_FLAGS, GL_CONTEXT_PROFILE_MASK,
468       GL_SHARE_WITH_CURRENT_CONTEXT,
469       GL_CONTEXT_RELEASE_BEHAVIOR,
470       GL_FRAMEBUFFER_SRGB_CAPABLE
471  
472     :const:`GL_CONTEXT_PROFILE_MASK`
473  
474       Sets the OpenGL profile to one of these values:
475  
476       ::
477  
478         GL_CONTEXT_PROFILE_CORE             disable deprecated features
479         GL_CONTEXT_PROFILE_COMPATIBILITY    allow deprecated features
480         GL_CONTEXT_PROFILE_ES               allow only the ES feature
481                                             subset of OpenGL
482  
483     :const:`GL_ACCELERATED_VISUAL`
484  
485       Set to 1 to require hardware acceleration, or 0 to force software render.
486       By default, both are allowed.

490  .. function:: get_active
491  
492     | :sl:`Returns True when the display is active on the screen`
493     | :sg:`get_active() -> bool`
494  
495     Returns True when the display Surface is considered actively
496     renderable on the screen and may be visible to the user.  This is
497     the default state immediately after ``pygame.display.set_mode()``.
498     This method may return True even if the application is fully hidden
499     behind another application window.
500  
501     This will return False if the display Surface has been iconified or
502     minimized (either via ``pygame.display.iconify()`` or via an OS
503     specific method such as the minimize-icon available on most
504     desktops).
505  
506     The method can also return False for other reasons without the
507     application being explicitly iconified or minimized by the user.  A
508     notable example being if the user has multiple virtual desktops and
509     the display Surface is not on the active virtual desktop.
510  
511     .. note:: This function returning True is unrelated to whether the
512         application has input focus.  Please see
513         ``pygame.key.get_focused()`` and ``pygame.mouse.get_focused()``
514         for APIs related to input focus.

534  .. function:: toggle_fullscreen
535  
536     | :sl:`Switch between fullscreen and windowed displays`
537     | :sg:`toggle_fullscreen() -> int`
538  
539     Switches the display window between windowed and fullscreen modes.
540     Display driver support is not great when using pygame 1, but with
541     pygame 2 it is the most reliable method to switch to and from fullscreen.
542  
543     Supported display drivers in pygame 1:
544  
545      * x11 (Linux/Unix)
546      * wayland (Linux/Unix)
547  
548     Supported display drivers in pygame 2:
549  
550      * windows (Windows)
551      * x11 (Linux/Unix)

553      * cocoa (OSX/Mac)
554  
555     .. Note:: :func:`toggle_fullscreen` doesn't work on Windows
556               unless the window size is in :func:`pygame.display.list_modes()` or
557               the window is created with the flag ``pygame.SCALED``.
558               See `issue #2380 <https://github.com/pygame/pygame/issues/2380>`_.

562  .. function:: set_gamma
563  
564     | :sl:`Change the hardware gamma ramps`
565     | :sg:`set_gamma(red, green=None, blue=None) -> bool`
566  
567     Set the red, green, and blue gamma values on the display hardware. If the
568     green and blue arguments are not passed, they will both be the same as red.
569     Not all systems and hardware support gamma ramps, if the function succeeds
570     it will return ``True``.
571  
572     A gamma value of ``1.0`` creates a linear color table. Lower values will
573     darken the display and higher values will brighten.

685  
686  .. function:: set_allow_screensaver
687  
688     | :sl:`Set whether the screensaver may run`
689     | :sg:`set_allow_screensaver(bool) -> None`
690  
691     Change whether screensavers should be allowed whilst the app is running.
692     The default value of the argument to the function is True.
693     By default pygame does not allow the screensaver during game play.
694  
695     If the screensaver has been disallowed due to this function, it will automatically
696     be allowed to run when :func:`pygame.quit()` is called.
697  
698     It is possible to influence the default value via the environment variable
699     ``SDL_HINT_VIDEO_ALLOW_SCREENSAVER``, which can be set to either ``0`` (disable)
700     or ``1`` (enable).
701  
702     .. note:: Disabling screensaver is subject to platform support.
703               When platform support is absent, this function will
704               silently appear to work even though the screensaver state
705               is unchanged.  The lack of feedback is due to SDL not
706               providing any supported method for determining whether
707               it supports changing the screensaver state.
708               ``SDL_HINT_VIDEO_ALLOW_SCREENSAVER`` is available in SDL 2.0.2 or later.
709               SDL1.2 does not implement this.
710
