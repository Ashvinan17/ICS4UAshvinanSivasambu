6    .. module:: pygame.event
7       :synopsis: pygame module for interacting with events and queues
8    
9    | :sl:`pygame module for interacting with events and queues`
10   
11   Pygame handles all its event messaging through an event queue. The routines in
12   this module help you manage that event queue. The input queue is heavily
13   dependent on the :mod:`pygame.display` module. If the display has not been
14   initialized and a video mode not set, the event queue may not work properly.
15   
16   The event queue has an upper limit on the number of events it can hold. When
17   the queue becomes full new events are quietly dropped. To prevent lost events,
18   especially input events which signal a quit command, your program must handle
19   events every frame (with ``pygame.event.get()``, ``pygame.event.pump()``,
20   ``pygame.event.wait()``, ``pygame.event.peek()`` or ``pygame.event.clear()``)
21   and process them. Not handling events may cause your system to decide your
22   program has locked up. To speed up queue processing use
23   :func:`pygame.event.set_blocked()` to limit which events get queued.
24   
25   To get the state of various input devices, you can forego the event queue and
26   access the input devices directly with their appropriate modules:
27   :mod:`pygame.mouse`, :mod:`pygame.key`, and :mod:`pygame.joystick`. If you use
28   this method, remember that pygame requires some form of communication with the
29   system window manager and other parts of the platform. To keep pygame in sync
30   with the system, you will need to call :func:`pygame.event.pump()` to keep
31   everything current. Usually, this should be called once per game loop.
32   Note: Joysticks will not send any events until the device has been initialized.
33   
34   The event queue contains :class:`pygame.event.EventType` event objects.
35   There are a variety of ways to access the queued events, from simply
36   checking for the existence of events, to grabbing them directly off the stack.
37   The event queue also offers some simple filtering which can slightly help
38   performance by blocking certain event types from the queue. Use
39   :func:`pygame.event.set_allowed()` and :func:`pygame.event.set_blocked()` to
40   change this filtering. By default, all event types can be placed on the queue.
41   
42   All :class:`pygame.event.EventType` instances contain an event type identifier
43   and attributes specific to that event type. The event type identifier is
44   accessible as the :attr:`pygame.event.EventType.type` property. Any of the
45   event specific attributes can be accessed through the
46   :attr:`pygame.event.EventType.__dict__` attribute or directly as an attribute
47   of the event object (as member lookups are passed through to the object's
48   dictionary values). The event object has no method functions. Users can create
49   their own new events with the :func:`pygame.event.Event()` function.
50   
51   The event type identifier is in between the values of ``NOEVENT`` and
52   ``NUMEVENTS``. User defined events should have a value in the inclusive range
53   of ``USEREVENT`` to ``NUMEVENTS - 1``. User defined events can get a custom
54   event number with :func:`pygame.event.custom_type()`. 
55   It is recommended all user events follow this system.
56   
57   Events support equality and inequality comparisons. Two events are equal if
58   they are the same type and have identical attribute values.
59   
60   While debugging and experimenting, you can print an event object for a quick
61   display of its type and members. The function :func:`pygame.event.event_name()`
62   can be used to get a string representing the name of the event type.
63   
64   Events that come from the system will have a guaranteed set of member
65   attributes based on the type. The following is a list event types with their
66   specific attributes.
67   
68   ::
69   
70       QUIT              none
71       ACTIVEEVENT       gain, state
72       KEYDOWN           key, mod, unicode, scancode
73       KEYUP             key, mod, unicode, scancode
74       MOUSEMOTION       pos, rel, buttons, touch
75       MOUSEBUTTONUP     pos, button, touch
76       MOUSEBUTTONDOWN   pos, button, touch
77       JOYAXISMOTION     joy (deprecated), instance_id, axis, value
78       JOYBALLMOTION     joy (deprecated), instance_id, ball, rel
79       JOYHATMOTION      joy (deprecated), instance_id, hat, value
80       JOYBUTTONUP       joy (deprecated), instance_id, button
81       JOYBUTTONDOWN     joy (deprecated), instance_id, button
82       VIDEORESIZE       size, w, h
83       VIDEOEXPOSE       none
84       USEREVENT         code

197  .. function:: pump
198  
199     | :sl:`internally process pygame event handlers`
200     | :sg:`pump() -> None`
201  
202     For each frame of your game, you will need to make some sort of call to the
203     event queue. This ensures your program can internally interact with the rest
204     of the operating system. If you are not using other event functions in your
205     game, you should call ``pygame.event.pump()`` to allow pygame to handle
206     internal actions.
207  
208     This function is not necessary if your program is consistently processing
209     events on the queue through the other :mod:`pygame.event` functions.
210  
211     There are important things that must be dealt with internally in the event
212     queue. The main window may need to be repainted or respond to the system. If
213     you fail to make a call to the event queue for too long, the system may
214     decide your program has locked up.
215  
216     .. caution::
217        This function should only be called in the thread that initialized :mod:`pygame.display`.
218

221  .. function:: get
222  
223     | :sl:`get events from the queue`
224     | :sg:`get(eventtype=None) -> Eventlist`
225     | :sg:`get(eventtype=None, pump=True) -> Eventlist`
226     | :sg:`get(eventtype=None, pump=True, exclude=None) -> Eventlist`
227  
228     This will get all the messages and remove them from the queue. If a type or
229     sequence of types is given only those messages will be removed from the
230     queue and returned.
231  
232     If a type or sequence of types is passed in the ``exclude`` argument
233     instead, then all only *other* messages will be removed from the queue. If
234     an ``exclude`` parameter is passed, the ``eventtype`` parameter *must* be
235     None.
236  
237     If you are only taking specific events from the queue, be aware that the
238     queue could eventually fill up with the events you are not interested.
239  
240     If ``pump`` is ``True`` (the default), then :func:`pygame.event.pump()` will be called.
241  
242     .. versionchanged:: 1.9.5 Added ``pump`` argument
243     .. versionchanged:: 2.0.2 Added ``exclude`` argument

261  .. function:: wait
262  
263     | :sl:`wait for a single event from the queue`
264     | :sg:`wait() -> EventType instance`
265     | :sg:`wait(timeout) -> EventType instance`
266  
267     Returns a single event from the queue. If the queue is empty this function
268     will wait until one is created. From pygame 2.0.0, if a ``timeout`` argument
269     is given, the function will return an event of type ``pygame.NOEVENT`` 
270     if no events enter the queue in ``timeout`` milliseconds. The event is removed
271     from the queue once it has been returned. While the program is waiting it will
272     sleep in an idle state. This is important for programs that want to share the
273     system with other applications.
274  
275     .. versionchanged:: 2.0.0.dev13 Added ``timeout`` argument
276  
277     .. caution::
278        This function should only be called in the thread that initialized :mod:`pygame.display`.

299  
300     | :sl:`remove all events from the queue`
301     | :sg:`clear(eventtype=None) -> None`
302     | :sg:`clear(eventtype=None, pump=True) -> None`
303  
304     Removes all events from the queue. If ``eventtype`` is given, removes the given event
305     or sequence of events. This has the same effect as :func:`pygame.event.get()` except ``None``

371  .. function:: set_grab
372  
373     | :sl:`control the sharing of input devices with other applications`
374     | :sg:`set_grab(bool) -> None`
375  
376     When your program runs in a windowed environment, it will share the mouse
377     and keyboard devices with other applications that have focus. If your
378     program sets the event grab to ``True``, it will lock all input into your
379     program.
380  
381     It is best to not always grab the input, since it prevents the user from
382     doing other things on their system.

395  .. function:: post
396  
397     | :sl:`place a new event on the queue`
398     | :sg:`post(Event) -> bool`
399  
400     Places the given event at the end of the event queue.
401  
402     This is usually used for placing custom events on the event queue.
403     Any type of event can be posted, and the events posted can have any attributes.
404  
405     This returns a boolean on whether the event was posted or not. Blocked events
406     cannot be posted, and this function returns ``False`` if you try to post them.
407  
408     .. versionchanged:: 2.0.1 returns a boolean, previously returned ``None``
