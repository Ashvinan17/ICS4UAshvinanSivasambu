101  <h2>Python Pygame Introduction<a class="headerlink" href="#python-pygame-introduction" title="Permalink to this headline">¶</a></h2>
102  <dl class="docinfo field-list simple">
103  <dt class="field-odd">Author</dt>
104  <dd class="field-odd"><p>Pete Shinners</p>
105  </dd>
106  <dt class="field-even">Contact</dt>
107  <dd class="field-even"><p><a class="reference external" href="mailto:pete&#37;&#52;&#48;shinners&#46;org">pete<span>&#64;</span>shinners<span>&#46;</span>org</a></p>
108  </dd>
109  </dl>
110  <p>This article is an introduction to the <a class="reference external" href="http://www.pygame.org">pygame library</a>
111  for <a class="reference external" href="https://www.python.org/">Python programmers</a>.
112  The original version appeared in the <a class="reference external" href="http://pyzine.com/?p=18">Py Zine</a>,
113  volume 1 issue 3. This version contains minor revisions, to
114  create an all-around better article. Pygame is a Python extension
115  library that wraps the <a class="reference external" href="http://www.libsdl.org">SDL</a> library
116  and its helpers.</p>
117  <section id="history">
118  <h3>HISTORY<a class="headerlink" href="#history" title="Permalink to this headline">¶</a></h3>
119  <p>Pygame started in the summer of 2000. Being a C programmer of many
120  years, I discovered both Python and SDL at about the same time. You are
121  already familiar with Python, which was at version 1.5.2. You may need
122  an introduction to SDL, which is the Simple DirectMedia Layer.
123  Created by Sam Lantinga, SDL is a cross-platform C library for
124  controlling multimedia, comparable to DirectX. It has been used for
125  hundreds of commercial and open source games. I was impressed at how clean
126  and straightforward both projects were and it wasn't long before I
127  realized mixing Python and SDL was an interesting proposal.</p>
128  <p>I discovered a small project already under-way with exactly the same
129  idea, PySDL. Created by Mark Baker, PySDL was a straightforward
130  implementation of SDL as a Python extension. The interface was cleaner
131  than a generic SWIG wrapping, but I felt it forced a &quot;C style&quot; of code.
132  The sudden death of PySDL prompted me to take on a new project of my
133  own.</p>
134  <p>I wanted to put together a project that really took advantage of
135  Python. My goal was to make it easy to do the simple things, and
136  straightforward to do the difficult things. Pygame was started in
137  October, 2000. Six months later pygame version 1.0 was released.</p>
138  </section>

174  <p>This is as simple as you can get for a bouncing animation.
175  First we see importing and initializing pygame is nothing noteworthy.
176  The <code class="docutils literal notranslate"><span class="pre">import</span> <span class="pre">pygame</span></code> imports the package with all the available
177  pygame modules.
178  The call to <code class="docutils literal notranslate"><span class="pre">pygame.init()</span></code> initializes each of these modules.</p>
179  <p>On <span class="codelineref">line 8</span> we create a
180  graphical window with the call to <code class="docutils literal notranslate"><span class="pre">pygame.display.set_mode()</span></code>.
181  Pygame and SDL make this easy by defaulting to the best graphics modes
182  for the graphics hardware. You can override the mode and SDL will
183  compensate for anything the hardware cannot do. Pygame represents
184  images as <em>Surface</em> objects.
185  The <code class="docutils literal notranslate"><span class="pre">display.set_mode()</span></code> function creates a new <em>Surface</em>
186  object that represents the actual displayed graphics. Any drawing you
187  do to this Surface will become visible on the monitor.</p>
188  <p>At <span class="codelineref">line 10</span> we load
189  our ball image. Pygame supports a variety of image formats through the
190  SDL_image library, including BMP, JPG, PNG, TGA, and GIF.
191  The <code class="docutils literal notranslate"><span class="pre">pygame.image.load()</span></code> function
192  returns us a Surface with the ball data. The Surface will keep any
193  colorkey or alpha transparency from the file. After loading the ball
194  image we create a variable named ballrect. Pygame comes with a
195  convenient utility object type named <a class="reference internal" href="../ref/rect.html#pygame.Rect" title="pygame.Rect"><code class="xref py py-class docutils literal notranslate"><span class="pre">Rect</span></code></a>,
196  which represents a rectangular area. Later, in the animation part of
197  the code, we will see what the <em>Rect</em> objects can do.</p>
198  <p>At this point, <span class="codelineref">line 13</span>,
199  our program is initialized and ready to run. Inside an infinite loop we
200  check for user input, move the ball, and then draw the ball. If you are
201  familiar with GUI programming, you have had experience with events and
202  event loops. In pygame this is no different,
203  we check if a <em>QUIT</em> event has happened. If so we
204  simply exit the program, pygame will ensure everything is cleanly
205  shutdown.</p>
206  <p>It is time to update our position for the ball.
207  <span class="codelineref">Lines 17</span> moves the ballrect variable by the current speed.
208  <span class="codelineref">Lines 18 thru 21</span> reverse the speed if the ball has moved outside the screen.
209  Not exactly Newtonian physics, but it is all we need.</p>
210  <p>On <span class="codelineref">line 23</span> we erase
211  the screen by filling it with a black RGB color. If you have never
212  worked with animations this may seem strange. You may be asking &quot;Why do
213  we need to erase anything, why don't we just move the ball on the
214  screen?&quot; That is not quite the way computer animation works. Animation
215  is nothing more than a series of single images, which when displayed in
216  sequence do a very good job of fooling the human eye into seeing
217  motion. The screen is just a single image that the user sees. If we did
218  not take the time to erase the ball from the screen, we would actually
219  see a &quot;trail&quot; of the ball as we continuously draw the ball in its new
220  positions.</p>
221  <p>On <span class="codelineref">line 24</span> we draw the ball image onto the screen.
222  Drawing of images is handled by the
223  <a class="reference internal" href="../ref/surface.html#pygame.Surface.blit" title="pygame.Surface.blit"><code class="xref py py-meth docutils literal notranslate"><span class="pre">Surface.blit()</span></code></a> method.
224  A blit basically means copying pixel colors from one image to another.
225  We pass the blit method a source <a class="reference internal" href="../ref/surface.html#pygame.Surface" title="pygame.Surface"><code class="xref py py-class docutils literal notranslate"><span class="pre">Surface</span></code></a>
226  to copy from, and a position to place the source onto the destination.</p>
227  <p>The last thing we need to do is actually update the visible display.
228  Pygame manages the display with a double buffer. When we are finished
229  drawing we call the <a class="tooltip reference internal" href="../ref/display.html#pygame.display.flip" title=""><code class="xref py py-func docutils literal notranslate"><span class="pre">pygame.display.flip()</span></code><span class="tooltip-content">Update the full display Surface to the screen</span></a> method.
230  This makes everything we have drawn on the screen Surface
231  become visible. This buffering makes sure we only see completely drawn
232  frames on the screen. Without it, the user would see the half completed
233  parts of the screen as they are being created.</p>
234  <p>That concludes this short introduction to pygame. Pygame also has
235  modules to do things like input handling for the keyboard, mouse, and
236  joystick. It can mix audio and decode streaming music.
237  With the <em>Surfaces</em> you can draw simple
238  shapes, rotate and scale the picture, and even manipulate the pixels of
239  an image in realtime as numpy arrays.
240  Pygame also has the ability to act as a
241  cross platform display layer for PyOpenGL. Most of the pygame modules
242  are written in C, few are actually done in Python.</p>
243  <p>The pygame website has full reference documentation for every pygame
244  function and tutorials for all ranges of users. The pygame source comes
245  with many examples of things like monkey punching and UFO shooting.</p>
246  </section>
247  <section id="python-and-gaming">
248  <h3>PYTHON AND GAMING<a class="headerlink" href="#python-and-gaming" title="Permalink to this headline">¶</a></h3>
249  <p>&quot;Is Python suitable for gaming?&quot; The answer is, &quot;It depends on the
250  game.&quot;</p>
251  <p>Python is actually quite capable at running games. It will likely even
252  surprise you how much is possible in under 30 milliseconds. Still, it
253  is not hard to reach the ceiling once your game begins to get more
254  complex. Any game running in realtime will be making full use of the
255  computer.</p>
256  <img alt="../_images/intro_blade.jpg" class="inlined-right" src="../_images/intro_blade.jpg" />
257  <p>Over the past several years there has been an interesting trend in game development,
258  the move towards higher level languages. Usually a game is split into
259  two major parts. The game engine, which must be as fast as possible,
260  and the game logic, which makes the engine actually do something. It
261  wasn't long ago when the engine of a game was written in assembly, with
262  portions written in C. Nowadays, C has moved to the game engine, while
263  often the game itself is written in higher level scripting languages.
264  Games like Quake3 and Unreal run these scripts as portable bytecode.</p>
265  <p>In early 2001, developer Rebel Act Studios finished their game,
266  Severance: Blade of Darkness. Using their own custom 3D engine, the
267  rest of the game is written with Python. The game is a bloody action
268  3rd person perspective fighter. You control medieval warriors into
269  intricate decapitating combination attacks while exploring dungeons and
270  castles. You can download third party add-ons for this game, and find
271  they are nothing more than Python source files.</p>
272  <p>More recently, Python has been used in a variety of games like Freedom
273  Force, and Humungous' Backyard Sports Series.</p>
274  <img alt="../_images/intro_freedom.jpg" class="inlined-right" src="../_images/intro_freedom.jpg" />
275  <p>Pygame and SDL serve as an excellent C engine for 2D games.
276  Games will still find the largest part of their runtime is spent
277  inside SDL handling the graphics.
278  SDL can take advantage of graphics hardware acceleration.
279  Enabling this can change a game from running around 40 frames per
280  second to over 200 frames per second. When you see your Python game
281  running at 200 frames per second, you realize that Python and games can
282  work together.</p>
283  <p>It is impressive how well both Python and SDL work on multiple
284  platforms. For example, in May of 2001 I released my own full pygame
285  project, SolarWolf, an arcade style action game. One thing that has
286  surprised me is that one year later there has been no need for any
287  patches, bug fixes, or updates. The game was developed entirely on
288  windows, but runs on Linux, Mac OSX, and many Unixes without any extra
289  work on my end.</p>
290  <p>Still, there are very clear limitations. The best way to manage
291  hardware accelerated graphics is not always the way to get fastest
292  results from software rendering. Hardware support is not available on
293  all platforms. When a game gets more complex, it often must commit to
294  one or the other. SDL has some other design limitations, things like
295  full screen scrolling graphics can quickly bring your game down to
296  unplayable speeds. While SDL is not suitable for all types of games,
297  remember companies like Loki have used SDL to run a wide variety of
298  retail quality titles.</p>
299  <p>Pygame is fairly low-level when it comes to writing games. You'll
300  quickly find yourself needing to wrap common functions into your own
301  game environment. The great thing about this is there is nothing inside
302  pygame to get in your way. Your program is in full control of
303  everything. The side effect of that is you will find yourself borrowing
304  a lot of code to get a more advanced framework put together. You'll
305  need a better understanding of what you are doing.</p>
306  </section>
307  <section id="closing">
308  <h3>CLOSING<a class="headerlink" href="#closing" title="Permalink to this headline">¶</a></h3>
309  <p>Developing games is very rewarding, there is something exciting about
310  being able to see and interact with the code you've written. Pygame
311  currently has almost 30 other projects using it. Several of them are
312  ready to play now. You may be surprised to visit the pygame website,
313  and see what other users have been able to do with Python.</p>
314  <p>One thing that has caught my attention is the amount of people coming
315  to Python for the first time to try game development. I can see why
316  games are a draw for new programmers, but it can be difficult since
317  creating games requires a firmer understanding of the language. I've
318  tried to support this group of users by writing many examples and
319  pygame tutorials for people new to these concepts.</p>
320  <p>In the end, my advice is to keep it simple. I cannot stress this
321  enough. If you are planning to create your first game, there is a
322  lot to learn. Even a simpler game will challenge your designs, and
323  complex games don't necessarily mean fun games. When you understand
324  Python, you can use pygame to create a simple game in only one or two
325  weeks. From there you'll need a surprising amount of time to add
326  the polish to make that into a full presentable game.</p>
