100  <section id="a-newbie-guide-to-pygame">
101  <h2>A Newbie Guide to pygame<a class="headerlink" href="#a-newbie-guide-to-pygame" title="Permalink to this headline">¶</a></h2>
102  <p>or <strong>Things I learned by trial and error so you don't have to,</strong></p>
103  <p>or <strong>How I learned to stop worrying and love the blit.</strong></p>
104  <p><a class="reference external" href="https://www.pygame.org/">Pygame</a> is a python wrapper for <a class="reference external" href="http://libsdl.org">SDL</a>, written by Pete Shinners.  What this
105  means is that, using pygame, you can write games or other multimedia
106  applications in Python that will run unaltered on any of SDL's supported
107  platforms (Windows, Unix, Mac, BeOS and others).</p>
108  <p>Pygame may be easy to learn, but the world of graphics programming can be
109  pretty confusing to the newcomer.  I wrote this to try to distill the practical
110  knowledge I've gained over the past year or so of working with pygame, and its
111  predecessor, PySDL.  I've tried to rank these suggestions in order of
112  importance, but how relevant any particular hint is will depend on your own
113  background and the details of your project.</p>
114  <section id="get-comfortable-working-in-python">
115  <h3>Get comfortable working in Python.<a class="headerlink" href="#get-comfortable-working-in-python" title="Permalink to this headline">¶</a></h3>
116  <p>The most important thing is to feel confident using python. Learning something
117  as potentially complicated as graphics programming will be a real chore if
118  you're also unfamiliar with the language you're using. Write a few sizable
119  non-graphical programs in python -- parse some text files, write a guessing
120  game or a journal-entry program or something. Get comfortable with string and
121  list manipulation -- know how to split, slice and combine strings and lists.
122  Know how <code class="docutils literal notranslate"><span class="pre">import</span></code> works -- try writing a program that is spread across
123  several source files.  Write your own functions, and practice manipulating
124  numbers and characters; know how to convert between the two.  Get to the point
125  where the syntax for using lists and dictionaries is second-nature -- you don't
126  want to have to run to the documentation every time you need to slice a list or
127  sort a set of keys.  Resist the temptation to run to a mailing list,
128  comp.lang.python, or IRC when you run into trouble.  Instead, fire up the
129  interpreter and play with the problem for a few hours.  Print out the <a class="reference external" href="http://www.brunningonline.net/simon/python/quick-ref2_0.html">Python
130  2.0 Quick Reference</a> and keep it by your computer.</p>
131  <p>This may sound incredibly dull, but the confidence you'll gain through your
132  familiarity with python will work wonders when it comes time to write your
133  game.  The time you spend making python code second-nature will be nothing
134  compared to the time you'll save when you're writing real code.</p>
135  </section>
136  <section id="recognize-which-parts-of-pygame-you-really-need">
137  <h3>Recognize which parts of pygame you really need.<a class="headerlink" href="#recognize-which-parts-of-pygame-you-really-need" title="Permalink to this headline">¶</a></h3>
138  <p>Looking at the jumble of classes at the top of the pygame Documentation index
139  may be confusing.  The important thing is to realize that you can do a great
140  deal with only a tiny subset of functions.  Many classes you'll probably never
141  use -- in a year, I haven't touched the <code class="docutils literal notranslate"><span class="pre">Channel</span></code>, <code class="docutils literal notranslate"><span class="pre">Joystick</span></code>, <code class="docutils literal notranslate"><span class="pre">cursors</span></code>,
142  <code class="docutils literal notranslate"><span class="pre">Userrect</span></code>, <code class="docutils literal notranslate"><span class="pre">surfarray</span></code> or <code class="docutils literal notranslate"><span class="pre">version</span></code> functions.</p>
143  </section>
144  <section id="know-what-a-surface-is">
145  <h3>Know what a surface is.<a class="headerlink" href="#know-what-a-surface-is" title="Permalink to this headline">¶</a></h3>
146  <p>The most important part of pygame is the surface.  Just think of a surface as a
147  blank piece of paper.  You can do a lot of things with a surface -- you can
148  draw lines on it, fill parts of it with color, copy images to and from it, and
149  set or read individual pixel colors on it.  A surface can be any size (within
150  reason) and you can have as many of them as you like (again, within reason).
151  One surface is special -- the one you create with
152  <code class="docutils literal notranslate"><span class="pre">pygame.display.set_mode()</span></code>.  This 'display surface' represents the screen;
153  whatever you do to it will appear on the user's screen.  You can only have one
154  of these -- that's an SDL limitation, not a pygame one.</p>
155  <p>So how do you create surfaces?  As mentioned above, you create the special
156  'display surface' with <code class="docutils literal notranslate"><span class="pre">pygame.display.set_mode()</span></code>.  You can create a surface
157  that contains an image by using <code class="docutils literal notranslate"><span class="pre">image.load()</span></code>, or you can make a surface
158  that contains text with <code class="docutils literal notranslate"><span class="pre">font.render()</span></code>.  You can even create a surface that
159  contains nothing at all with <code class="docutils literal notranslate"><span class="pre">Surface()</span></code>.</p>
160  <p>Most of the surface functions are not critical. Just learn <code class="docutils literal notranslate"><span class="pre">blit()</span></code>,
161  <code class="docutils literal notranslate"><span class="pre">fill()</span></code>, <code class="docutils literal notranslate"><span class="pre">set_at()</span></code> and <code class="docutils literal notranslate"><span class="pre">get_at()</span></code>, and you'll be fine.</p>
162  </section>
163  <section id="use-surface-convert">
164  <h3>Use surface.convert().<a class="headerlink" href="#use-surface-convert" title="Permalink to this headline">¶</a></h3>
165  <p>When I first read the documentation for <code class="docutils literal notranslate"><span class="pre">surface.convert()</span></code>, I didn't think
166  it was something I had to worry about. 'I only use PNGs, therefore everything I
167  do will be in the same format. So I don't need <code class="docutils literal notranslate"><span class="pre">convert()</span></code>';. It turns out I
168  was very, very wrong.</p>
169  <p>The 'format' that <code class="docutils literal notranslate"><span class="pre">convert()</span></code> refers to isn't the <em>file</em> format (ie PNG,
170  JPEG, GIF), it's what's called the 'pixel format'.  This refers to the
171  particular way that a surface records individual colors in a specific pixel.
172  If the surface format isn't the same as the display format, SDL will have to
173  convert it on-the-fly for every blit -- a fairly time-consuming process.  Don't
174  worry too much about the explanation; just note that <code class="docutils literal notranslate"><span class="pre">convert()</span></code> is necessary
175  if you want to get any kind of speed out of your blits.</p>
176  <p>How do you use convert? Just call it after creating a surface with the
177  <code class="docutils literal notranslate"><span class="pre">image.load()</span></code> function. Instead of just doing:</p>
178  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="n">surface</span> <span class="o">=</span> <span class="n">pygame</span><span class="o">.</span><span class="n">image</span><span class="o">.</span><span class="n">load</span><span class="p">(</span><span class="s1">&#39;foo.png&#39;</span><span class="p">)</span>
179  </pre></div>
180  </div>
181  <p>Do:</p>
182  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="n">surface</span> <span class="o">=</span> <span class="n">pygame</span><span class="o">.</span><span class="n">image</span><span class="o">.</span><span class="n">load</span><span class="p">(</span><span class="s1">&#39;foo.png&#39;</span><span class="p">)</span><span class="o">.</span><span class="n">convert</span><span class="p">()</span>
183  </pre></div>
184  </div>
185  <p>It's that easy. You just need to call it once per surface, when you load an
186  image off the disk.  You'll be pleased with the results; I see about a 6x
187  increase in blitting speed by calling <code class="docutils literal notranslate"><span class="pre">convert()</span></code>.</p>
188  <p>The only times you don't want to use <code class="docutils literal notranslate"><span class="pre">convert()</span></code> is when you really need to
189  have absolute control over an image's internal format -- say you were writing
190  an image conversion program or something, and you needed to ensure that the
191  output file had the same pixel format as the input file.  If you're writing a
192  game, you need speed.  Use <code class="docutils literal notranslate"><span class="pre">convert()</span></code>.</p>
193  </section>
194  <section id="dirty-rect-animation">
195  <h3>Dirty rect animation.<a class="headerlink" href="#dirty-rect-animation" title="Permalink to this headline">¶</a></h3>
196  <p>The most common cause of inadequate frame rates in pygame programs results from
197  misunderstanding the <code class="docutils literal notranslate"><span class="pre">pygame.display.update()</span></code> function.  With pygame, merely
198  drawing something to the display surface doesn't cause it to appear on the
199  screen -- you need to call <code class="docutils literal notranslate"><span class="pre">pygame.display.update()</span></code>.  There are three ways
200  of calling this function:</p>
201  <blockquote>
202  <div><ul class="simple">
203  <li><p><code class="docutils literal notranslate"><span class="pre">pygame.display.update()</span></code> -- This updates the whole window (or the whole screen for fullscreen displays).</p></li>
204  <li><p><code class="docutils literal notranslate"><span class="pre">pygame.display.flip()</span></code> -- This does the same thing, and will also do the right thing if you're using <code class="docutils literal notranslate"><span class="pre">double-buffered</span></code> hardware acceleration, which you're not, so on to...</p></li>
205  <li><p><code class="docutils literal notranslate"><span class="pre">pygame.display.update(a</span> <span class="pre">rectangle</span> <span class="pre">or</span> <span class="pre">some</span> <span class="pre">list</span> <span class="pre">of</span> <span class="pre">rectangles)</span></code> -- This updates just the rectangular areas of the screen you specify.</p></li>
206  </ul>
207  </div></blockquote>
208  <p>Most people new to graphics programming use the first option -- they update the
209  whole screen every frame.  The problem is that this is unacceptably slow for
210  most people.  Calling <code class="docutils literal notranslate"><span class="pre">update()</span></code> takes 35 milliseconds on my machine, which
211  doesn't sound like much, until you realize that 1000 / 35 = 28 frames per
212  second <em>maximum</em>. And that's with no game logic, no blits, no input, no AI,
213  nothing.  I'm just sitting there updating the screen, and 28 fps is my maximum
214  framerate. Ugh.</p>
215  <p>The solution is called 'dirty rect animation'.  Instead of updating the whole
216  screen every frame, only the parts that changed since the last frame are
217  updated.  I do this by keeping track of those rectangles in a list, then
218  calling <code class="docutils literal notranslate"><span class="pre">update(the_dirty_rectangles)</span></code> at the end of the frame.  In detail
219  for a moving sprite, I:</p>
220  <blockquote>
221  <div><ul class="simple">
222  <li><p>Blit a piece of the background over the sprite's current location, erasing it.</p></li>
223  <li><p>Append the sprite's current location rectangle to a list called dirty_rects.</p></li>
224  <li><p>Move the sprite.</p></li>
225  <li><p>Draw the sprite at its new location.</p></li>
226  <li><p>Append the sprite's new location to my dirty_rects list.</p></li>
227  <li><p>Call <code class="docutils literal notranslate"><span class="pre">display.update(dirty_rects)</span></code></p></li>
228  </ul>
229  </div></blockquote>
230  <p>The difference in speed is astonishing. Consider that <a class="reference external" href="https://www.pygame.org/shredwheat/solarwolf/index.shtml">SolarWolf</a> has dozens of
231  constantly moving sprites updating smoothly, and still has enough time left
232  over to display a parallax starfield in the background, and update that too.</p>
233  <p>There are two cases where this technique just won't work. The first is where
234  the whole window or screen really is being updated every frame -- think of a
235  smooth-scrolling engine like an overhead real-time strategy game or a
236  side-scroller.  So what do you do in this case?  Well, the short answer is --
237  don't write this kind of game in pygame.  The long answer is to scroll in steps
238  of several pixels at a time; don't try to make scrolling perfectly smooth.
239  Your player will appreciate a game that scrolls quickly, and won't notice the
240  background jumping along too much.</p>
241  <p>A final note -- not every game requires high framerates. A strategic wargame
242  could easily get by on just a few updates per second -- in this case, the added
243  complexity of dirty rect animation may not be necessary.</p>
244  </section>

248  <section id="hardware-surfaces-are-more-trouble-than-they-re-worth">
249  <h3>Hardware surfaces are more trouble than they're worth.<a class="headerlink" href="#hardware-surfaces-are-more-trouble-than-they-re-worth" title="Permalink to this headline">¶</a></h3>
250  <p><strong>Especially in pygame 2, because HWSURFACE now does nothing</strong></p>
251  <p>If you've been looking at the various flags you can use with
252  <code class="docutils literal notranslate"><span class="pre">pygame.display.set_mode()</span></code>, you may have thought like this: <cite>Hey,
253  HWSURFACE! Well, I want that -- who doesn't like hardware acceleration. Ooo...
254  DOUBLEBUF; well, that sounds fast, I guess I want that too!</cite>.  It's not
255  your fault; we've been trained by years of 3-d gaming to believe that hardware
256  acceleration is good, and software rendering is slow.</p>
257  <p>Unfortunately, hardware rendering comes with a long list of drawbacks:</p>
258  <blockquote>
259  <div><ul class="simple">
260  <li><p>It only works on some platforms. Windows machines can usually get hardware surfaces if you ask for them. Most other platforms can't. Linux, for example, may be able to provide a hardware surface if X4 is installed, if DGA2 is working properly, and if the moons are aligned correctly. If a hardware surface is unavailable, SDL will silently give you a software surface instead.</p></li>
261  <li><p>It only works fullscreen.</p></li>
262  <li><p>It complicates per-pixel access.  If you have a hardware surface, you need to Lock the surface before writing or reading individual pixel values on it.  If you don't, Bad Things Happen. Then you need to quickly Unlock the surface again, before the OS gets all confused and starts to panic.  Most of this process is automated for you in pygame, but it's something else to take into account.</p></li>
263  <li><p>You lose the mouse pointer. If you specify <code class="docutils literal notranslate"><span class="pre">HWSURFACE</span></code> (and actually get it), your pointer will usually just vanish (or worse, hang around in a half-there, half-not flickery state).  You'll need to create a sprite to act as a manual mouse pointer, and you'll need to worry about pointer acceleration and sensitivity. What a pain.</p></li>
264  <li><p>It might be slower anyway. Many drivers are not accelerated for the types of drawing that we do, and since everything has to be blitted across the video bus (unless you can cram your source surface into video memory as well), it might end up being slower than software access anyway.</p></li>
265  </ul>
266  </div></blockquote>
267  <p>Hardware rendering has its place. It works pretty reliably under Windows, so
268  if you're not interested in cross-platform performance, it may provide you with
269  a substantial speed increase.  However, it comes at a cost -- increased
270  headaches and complexity.  It's best to stick with good old reliable
271  <code class="docutils literal notranslate"><span class="pre">SWSURFACE</span></code> until you're sure you know what you're doing.</p>
272  </section>
273  <section id="don-t-get-distracted-by-side-issues">
274  <h3>Don't get distracted by side issues.<a class="headerlink" href="#don-t-get-distracted-by-side-issues" title="Permalink to this headline">¶</a></h3>
275  <p>Sometimes, new game programmers spend too much time worrying about issues that
276  aren't really critical to their game's success.  The desire to get secondary
277  issues 'right' is understandable, but early in the process of creating a game,
278  you cannot even know what the important questions are, let alone what answers
279  you should choose.  The result can be a lot of needless prevarication.</p>
280  <p>For example, consider the question of how to organize your graphics files.
281  Should each frame have its own graphics file, or each sprite?  Perhaps all the
282  graphics should be zipped up into one archive?  A great deal of time has been
283  wasted on a lot of projects, asking these questions on mailing lists, debating
284  the answers, profiling, etc, etc.  This is a secondary issue; any time spent
285  discussing it should have been spent coding the actual game.</p>
286  <p>The insight here is that it is far better to have a 'pretty good' solution that
287  was actually implemented, than a perfect solution that you never got around to
288  writing.</p>
289  </section>
290  <section id="rects-are-your-friends">
291  <h3>Rects are your friends.<a class="headerlink" href="#rects-are-your-friends" title="Permalink to this headline">¶</a></h3>
292  <p>Pete Shinners' wrapper may have cool alpha effects and fast blitting speeds,
293  but I have to admit my favorite part of pygame is the lowly <code class="docutils literal notranslate"><span class="pre">Rect</span></code> class.  A
294  rect is simply a rectangle -- defined only by the position of its top left
295  corner, its width, and its height.  Many pygame functions take rects as
296  arguments, and they also take 'rectstyles', a sequence that has the same values
297  as a rect. So if I need a rectangle that defines the area between 10, 20 and
298  40, 50, I can do any of the following:</p>
299  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="n">rect</span> <span class="o">=</span> <span class="n">pygame</span><span class="o">.</span><span class="n">Rect</span><span class="p">(</span><span class="mi">10</span><span class="p">,</span> <span class="mi">20</span><span class="p">,</span> <span class="mi">30</span><span class="p">,</span> <span class="mi">30</span><span class="p">)</span>
300  <span class="n">rect</span> <span class="o">=</span> <span class="n">pygame</span><span class="o">.</span><span class="n">Rect</span><span class="p">((</span><span class="mi">10</span><span class="p">,</span> <span class="mi">20</span><span class="p">,</span> <span class="mi">30</span><span class="p">,</span> <span class="mi">30</span><span class="p">))</span>
301  <span class="n">rect</span> <span class="o">=</span> <span class="n">pygame</span><span class="o">.</span><span class="n">Rect</span><span class="p">((</span><span class="mi">10</span><span class="p">,</span> <span class="mi">20</span><span class="p">),</span> <span class="p">(</span><span class="mi">30</span><span class="p">,</span> <span class="mi">30</span><span class="p">))</span>
302  <span class="n">rect</span> <span class="o">=</span> <span class="p">(</span><span class="mi">10</span><span class="p">,</span> <span class="mi">20</span><span class="p">,</span> <span class="mi">30</span><span class="p">,</span> <span class="mi">30</span><span class="p">)</span>
303  <span class="n">rect</span> <span class="o">=</span> <span class="p">((</span><span class="mi">10</span><span class="p">,</span> <span class="mi">20</span><span class="p">,</span> <span class="mi">30</span><span class="p">,</span> <span class="mi">30</span><span class="p">))</span>
304  </pre></div>
305  </div>
306  <p>If you use any of the first three versions, however, you get access to Rect's
307  utility functions.  These include functions to move, shrink and inflate rects,
308  find the union of two rects, and a variety of collision-detection functions.</p>
309  <p>For example, suppose I'd like to get a list of all the sprites that contain a
310  point (x, y) -- maybe the player clicked there, or maybe that's the current
311  location of a bullet. It's simple if each sprite has a .rect member -- I just
312  do:</p>
313  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="n">sprites_clicked</span> <span class="o">=</span> <span class="p">[</span><span class="n">sprite</span> <span class="k">for</span> <span class="n">sprite</span> <span class="ow">in</span> <span class="n">all_my_sprites_list</span> <span class="k">if</span> <span class="n">sprite</span><span class="o">.</span><span class="n">rect</span><span class="o">.</span><span class="n">collidepoint</span><span class="p">(</span><span class="n">x</span><span class="p">,</span> <span class="n">y</span><span class="p">)]</span>
314  </pre></div>
315  </div>
316  <p>Rects have no other relation to surfaces or graphics functions, other than the
317  fact that you can use them as arguments.  You can also use them in places that
318  have nothing to do with graphics, but still need to be defined as rectangles.
319  Every project I discover a few new places to use rects where I never thought
320  I'd need them.</p>
321  </section>
322  <section id="don-t-bother-with-pixel-perfect-collision-detection">
323  <h3>Don't bother with pixel-perfect collision detection.<a class="headerlink" href="#don-t-bother-with-pixel-perfect-collision-detection" title="Permalink to this headline">¶</a></h3>
324  <p>So you've got your sprites moving around, and you need to know whether or not they're bumping into one another. It's tempting to write something like the following:</p>
325  <blockquote>
326  <div><ul class="simple">
327  <li><p>Check to see if the rects are in collision. If they aren't, ignore them.</p></li>
328  <li><p>For each pixel in the overlapping area, see if the corresponding pixels from both sprites are opaque. If so, there's a collision.</p></li>
329  </ul>
330  </div></blockquote>
331  <p>There are other ways to do this, with ANDing sprite masks and so on, but any
332  way you do it in pygame, it's probably going to be too slow. For most games,
333  it's probably better just to do 'sub-rect collision' -- create a rect for each
334  sprite that's a little smaller than the actual image, and use that for
335  collisions instead. It will be much faster, and in most cases the player won't
336  notice the imprecision.</p>
337  </section>
338  <section id="managing-the-event-subsystem">
339  <h3>Managing the event subsystem.<a class="headerlink" href="#managing-the-event-subsystem" title="Permalink to this headline">¶</a></h3>
340  <p>Pygame's event system is kind of tricky.  There are actually two different ways
341  to find out what an input device (keyboard, mouse or joystick) is doing.</p>
342  <p>The first is by directly checking the state of the device.  You do this by
343  calling, say, <code class="docutils literal notranslate"><span class="pre">pygame.mouse.get_pos()</span></code> or <code class="docutils literal notranslate"><span class="pre">pygame.key.get_pressed()</span></code>.
344  This will tell you the state of that device <em>at the moment you call the
345  function.</em></p>
346  <p>The second method uses the SDL event queue.  This queue is a list of events --
347  events are added to the list as they're detected, and they're deleted from the
348  queue as they're read off.</p>
349  <p>There are advantages and disadvantages to each system.  State-checking (system
350  1) gives you precision -- you know exactly when a given input was made -- if
351  <code class="docutils literal notranslate"><span class="pre">mouse.get_pressed([0])</span></code> is 1, that means that the left mouse button is
352  down <em>right at this moment</em>.  The event queue merely reports that the
353  mouse was down at some time in the past; if you check the queue fairly often,
354  that can be ok, but if you're delayed from checking it by other code, input
355  latency can grow.  Another advantage of the state-checking system is that it
356  detects &quot;chording&quot; easily; that is, several states at the same time.  If you
357  want to know whether the <code class="docutils literal notranslate"><span class="pre">t</span></code> and <code class="docutils literal notranslate"><span class="pre">f</span></code> keys are down at the same time, just
358  check:</p>
359  <div class="highlight-default notranslate"><div class="highlight"><pre><span></span><span class="k">if</span> <span class="p">(</span><span class="n">key</span><span class="o">.</span><span class="n">get_pressed</span><span class="p">[</span><span class="n">K_t</span><span class="p">]</span> <span class="ow">and</span> <span class="n">key</span><span class="o">.</span><span class="n">get_pressed</span><span class="p">[</span><span class="n">K_f</span><span class="p">]):</span>
360      <span class="nb">print</span> <span class="s2">&quot;Yup!&quot;</span>
361  </pre></div>
362  </div>
363  <p>In the queue system, however, each keypress arrives in the queue as a
364  completely separate event, so you'd need to remember that the <code class="docutils literal notranslate"><span class="pre">t</span></code> key was
365  down, and hadn't come up yet, while checking for the <code class="docutils literal notranslate"><span class="pre">f</span></code> key.  A little more
366  complicated.</p>
367  <p>The state system has one great weakness, however. It only reports what the
368  state of the device is at the moment it's called; if the user hits a mouse
369  button then releases it just before a call to <code class="docutils literal notranslate"><span class="pre">mouse.get_pressed()</span></code>, the
370  mouse button will return 0 -- <code class="docutils literal notranslate"><span class="pre">get_pressed()</span></code> missed the mouse button press
371  completely.  The two events, <code class="docutils literal notranslate"><span class="pre">MOUSEBUTTONDOWN</span></code> and <code class="docutils literal notranslate"><span class="pre">MOUSEBUTTONUP</span></code>, will
372  still be sitting in the event queue, however, waiting to be retrieved and
373  processed.</p>
374  <p>The lesson is: choose the system that meets your requirements.  If you don't
375  have much going on in your loop -- say you're just sitting in a <code class="docutils literal notranslate"><span class="pre">while</span> <span class="pre">1</span></code>
376  loop, waiting for input, use <code class="docutils literal notranslate"><span class="pre">get_pressed()</span></code> or another state function; the
377  latency will be lower.  On the other hand, if every keypress is crucial, but
378  latency isn't as important -- say your user is typing something in an editbox,
379  use the event queue.  Some keypresses may be slightly late, but at least you'll
380  get them all.</p>
381  <p>A note about <code class="docutils literal notranslate"><span class="pre">event.poll()</span></code> vs. <code class="docutils literal notranslate"><span class="pre">wait()</span></code> -- <code class="docutils literal notranslate"><span class="pre">poll()</span></code> may seem better,
382  since it doesn't block your program from doing anything while it's waiting for
383  input -- <code class="docutils literal notranslate"><span class="pre">wait()</span></code> suspends the program until an event is received.
384  However, <code class="docutils literal notranslate"><span class="pre">poll()</span></code> will consume 100% of available CPU time while it runs,
385  and it will fill the event queue with <code class="docutils literal notranslate"><span class="pre">NOEVENTS</span></code>.  Use <code class="docutils literal notranslate"><span class="pre">set_blocked()</span></code> to
386  select just those event types you're interested in -- your queue will be much
387  more manageable.</p>
388  </section>
389  <section id="colorkey-vs-alpha">
390  <h3>Colorkey vs. Alpha.<a class="headerlink" href="#colorkey-vs-alpha" title="Permalink to this headline">¶</a></h3>
391  <p>There's a lot of confusion around these two techniques, and much of it comes from the terminology used.</p>
392  <p>'Colorkey blitting' involves telling pygame that all pixels of a certain color
393  in a certain image are transparent instead of whatever color they happen to be.
394  These transparent pixels are not blitted when the rest of the image is blitted,
395  and so don't obscure the background.  This is how we make sprites that aren't
396  rectangular in shape.  Simply call <code class="docutils literal notranslate"><span class="pre">surface.set_colorkey(color)</span></code>, where
397  color is an RGB tuple -- say (0,0,0). This would make every pixel in the source
398  image transparent instead of black.</p>
399  <p>'Alpha' is different, and it comes in two flavors. 'Image alpha' applies to the
400  whole image, and is probably what you want.  Properly known as 'translucency',
401  alpha causes each pixel in the source image to be only <em>partially</em> opaque.
402  For example, if you set a surface's alpha to 192 and then blitted it onto a
403  background, 3/4 of each pixel's color would come from the source image, and 1/4
404  from the background.  Alpha is measured from 255 to 0, where 0 is completely
405  transparent, and 255 is completely opaque.  Note that colorkey and alpha
406  blitting can be combined -- this produces an image that is fully transparent in
407  some spots, and semi-transparent in others.</p>
408  <p>'Per-pixel alpha' is the other flavor of alpha, and it's more complicated.
409  Basically, each pixel in the source image has its own alpha value, from 0 to
410  255.  Each pixel, therefore, can have a different opacity when blitted onto a
411  background.  This type of alpha can't be mixed with colorkey blitting,
412  and it overrides per-image alpha.  Per-pixel alpha is rarely used in
413  games, and to use it you have to save your source image in a graphic
414  editor with a special <em>alpha channel</em>.  It's complicated -- don't use it
415  yet.</p>
416  </section>
417  <section id="do-things-the-pythony-way">
418  <h3>Do things the pythony way.<a class="headerlink" href="#do-things-the-pythony-way" title="Permalink to this headline">¶</a></h3>
419  <p>A final note (this isn't the least important one; it just comes at the end).
420  Pygame is a pretty lightweight wrapper around SDL, which is in turn a pretty
421  lightweight wrapper around your native OS graphics calls.  Chances are pretty
422  good that if your code is still slow, and you've done the things I've mentioned
423  above, then the problem lies in the way you're addressing your data in python.
424  Certain idioms are just going to be slow in python no matter what you do.
425  Luckily, python is a very clear language -- if a piece of code looks awkward or
426  unwieldy, chances are its speed can be improved, too.  Read over <a class="reference external" href="http://www-rohan.sdsu.edu/~gawron/compling/course_core/python_intro/intro_lecture_files/fastpython.html">Python
427  Performance Tips</a> for some great advice on how you can improve the speed of
428  your code.  That said, premature optimisation is the root of all evil; if it's
429  just not fast enough, don't torture the code trying to make it faster.  Some
430  things are just not meant to be :)</p>
431  <p>There you go. Now you know practically everything I know about using pygame.
432  Now, go write that game!</p>
